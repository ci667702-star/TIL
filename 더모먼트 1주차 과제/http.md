# HTTP(HyperText Transfer Protocol)

## HTTP(HyperText Transfer Protocol)의 개념과 역할

**HTTP의 의미**
HTTP는 월드 와이드 웹(World Wide Web, WWW)은 통신하는 데 사용하는 프로토콜이다. HTTP는 다양한 종류의 데이터를 전송할 수 있도록 설계되었다. 일반적으로 TCP/IP 통신 프로토콜을 기반으로 동작한다.

**특징**
- 클라이언트와 서버간의 요청과 응답을 통해 작동한다.
- 다양한 종류의 데이터(html, css, javascript, png, gif 등)를 전송할 수 있도록 설계된 프로토콜이다.
- 웹페이지에 포함된 각 구성 요소(이미지, 텍스트, html, 영상 등)는 대부분 개별적인 HTTP 요청을 통해 서버로부터 요청된다.
- 일반적으로 TCP/IP 프로토콜을 기반으로 동작한다.

**장점**
1. 서버 확장성이 높다
    - Stateless라서 응답서버를 쉽게 바꿀 수도 있고 무한한 서버 증설이 가능하다.
2. 트래픽이 많지 않고 빠른 응답을 제공할 수 있는 경우 효율적이다.
3. 최소한의 자원으로 서버를 유지할 수 있다.

**단점**
1. 이전 통신의 정보를 모르기 때문에 매번 인증을 해줘야 한다.
2. 이것을 해결하기 위해 쿠키나 세션을 사용해 데이터를 처리해주어야 한다.
3. 보안이 되어있지 않다.
4. 매번 새로 연결해야하므로 3way-handshake 시간이 추가된다.

**HTTPS와의 차이**
구분|HTTP | HTTPS
:--:|:--:|:--:
프로토콜 | HyperText Transfer Protocol | HTTP + Secure(SSL/TLS)
암호화 |X|SSL 또는 TLS를 사용하여 암호화
포트 번호 | 80 | 443
인증서 필요성 | 필요 없음 | SSL/TLS 인증서 필요
URL 표기 | http:// | https://

**기반 프로토콜**
- TCP: HTTP/1.1, HTTP/2
- UDP: HTTP/3

## Stateless 구조와 클라이언트-서버 구조
**Stateless 구조**<br>
HTTP는 기본적으로 상태를 저장하지 않기 때문에 클라이언트와 서버 간의 요청은 독립적으로 처리된다.<br>

*특징*
- 각 요청에 필요한 모든 정보를 포함시켜야한다.
- 서버는 이전 요청과 관계 없이 처리하므로 단순성과 확장성이 높다.
- 상태 저장이 필요하다면 클라이언트가 정보를 저장하고 요청에 포함한다.

*장점*
- 서버의 부하가 적어 확장성이 뛰어나다.
- 설계와 구현이 상대적으로 간단하다.

*단점*
- 클라이언트가 요청마다 필요한 데이터를 전달해야 하므로 데이터 오버헤드가 발생할 수 있다.
- 연속성이 필요한 작업에는 적합하지 않다.

#### Stateful과 Stateless 비교
구분 | Stateful | Stateless
:--:|:--|:--
상태 유지 여부 | O | X
서버 리소스 소모 | 높음 | 낮음
확장성 | 제한적 | 우수


**클라이언트-서버 구조(Request Response 구조)**<br>
클라이언트가 요청을 보내면 서버가 응답을 보낸다.
이렇게 클라이언트와 서버를 분리함으로써 각각의 역할에 맞는 책임만 수행하면 되기에 각 역할은 자신의 책임에 집중할 수 있다.

**특징**
- 요청-응답의 형태
    - 클라이언트 서버 구조에서의 통신은 항상 요청-응답 형태이기 때문에 클라이언트의 요청으로만 통신이 시작된다.
- 동시 요구 처리
    - 서버는 여러 다른 클라이언트들의 동시적인 요청을 처리할 수 있도록 설계가 되어있다.
- 구성 요소
    - 클라이언트 서버 구조의 구성 요소로는 클라이언트, 서버, 그리고 네트워크가 있다.
- 중앙 집중화
    - 서버는 데이터를 중앙에서 관리하며, 이는 데이터의 일관성을 유지하도록 해주는 동시에 접근 제어 및 암호화를 통해 데이터를 보호할 수 있다.


## HTTP 메서드
**GET**<br>
가장 흔히 쓰이는 메서드이다. 주로 서버에게 리소스를 달라고 요청하기 위해 쓰인다. 

**POST**<br>
서버에 입력 데이터를 전송하기 위해 설계되었다. 실제로 HTML 폼을 지원하기 위해 흔히 사용된다. 채워진 에 담긴 데이터는 서버로폼 전송되며, 서버는 이를 모아서 필요로 하는 곳에 보낸다. 

**PUT**<br>
GET 메서드가 서버로부터 문서를 읽어 들이는데 반해 PUT 메서드는 서버에 문서를 쓴다. 어떤 발행 시스템은 사용자가 PUT을 이용해 웹페이지를 만들고 웹 서버에 직접 게시할 수 있도록 한다. PUT 메서드의 의미는, 서버가 요청의 본문을 가지고 요청 URL의 이름대로 새 문서를 만들거나. 이미 URL이 존재한다면 본문을 사용해서 교체하는 것이다.
즉, PUT은 콘텐츠를 변경할 수 있게 해준다.

**DELETE**<br>
서버에게 요청 URL로 지정한 리소스를 삭제할 것을 요청한다. 그러나 클라이언트는 삭제가 수행되는 것을 보지 못한다. 왜냐하면 HTTP 명세는 서버가 클라이언트에게 알리지 않고 요청을 무시하는 것을 허용하기 때문이다.


메서드 | 설명 | 메세지 본문이 있는가?
:-- | :-- | :--
GET | 서버에서 어떤 문서를 가져온다. | 없음
POST | 서버가 처리해야 할 데이터를 보낸다. | 있음
PUT | 서버에 요청 메세지의 본문을 저장한다. | 있음
DELETE | 서버에서 문서를 제거한다. | 없음 

## HTTP 상태 코드
**HTTP 상태 코드의 의미**<br>
상태 코드는 클라이언트에게 요청이 성공했는지 아니면 추가 조치가 필요한지 알려주는 세 자리 숫자이다.이 HTTP 상태코드를 통해 사용자는 클라이언트와 서버 간의 통신 과정에서 어떤 일이 발생했는지를 대략적으로 유추할 수 있다. <br><br>

### 흔히 쓰이는 상태 코드

**1. 200번대: 처리 완료(Successful)**
HTTP 상태코드|사유구절|설명
:--:|:--|:--
200| OK|클라이언트의 요청을 서버가 성공적으로 처리.
201 |Created| 클라이언트의 요청을 서버가 성공적으로 받아들여 새로운 리소스 생성.<br> 일반적으로 POST 요청 또는 일부 PUT 요청 이후에 따라옴.
202 |Accepted | 클라이언트의 요청을 성공적으로 접수되었으나 처리가 완료되지는 않음.<br> 주로 서버가 요청을 처리 혹은 대기하고 있거나 다른 프로세스에서 처리되는 경우에 쓰임.
204 |No Content| 성공적으로 수행했지만(오류는 없지만), 처리할 데이터가 없음.

200번대는 성공적으로 처리되었음을 뜻하는 상태 코드이다. 다만 204와 같은 코드는 디버깅 시 유의해서 봐야한다.<br><br>

**2. 300번대: 추가 조치 필요(Redirection)**

HTTP 상태코드|사유구절|설명
:--:|:--|:--
300| Multiple Choice | 클라이언트에게 여러 선택사항이 있음을 나타냄. <br>하나 이상의 응답을 선택할 수 있으며 반드시 하나를 선택하여 다시 요청해야함.
301 |Moved Permanently|새로운 위치로 옮겨지는 등의 이유로 요청한 리소스의 URI가 영구히 변경되었음을 의미함. <br>클라이언트는 새로운 리소스 URI를 다시 요청해야함.
302 |Found | 요청한 리소스의 URI가 일시적으로 변경되었음을 의미함.
308 |Permanent Redirect| 변경된 URI로 리디렉션 요청.<br>사용자가 반드시 사용된 HTTP 메서드(GET,POST)를 변경하지 않고 그대로 보내야함.

(영구적인 리다이렉트: URI 영구변경)<br>

HTTP 상태코드|사유구절|설명
:--:|:--|:--
302 |Found| 리다이렉트 요청 시, GET으로 변경될 수 있음
303| See Other| 리다이렉트 요청 시, GET으로 변경
307 |Temporary Redirect| 변경된 URI로 리디렉션 요청
(일시적인 리다이렉트:URI 일시변경)<br>


HTTP 상태코드|사유구절|설명
:--:|:--|:--
304| Not Modified| 클라이언트는 헤더를 이용해 조건부 요청을 만들 수 있음.<br> 만약 클라이언트가 GET과 같은 조건부 요청을 보냈고 그 요청한 리소스가 최근에 수정된 일이 없다면 이 코드는 리소스가 수정되지 않았음을 의미하게 된다.
(캐시 재사용)<br><br>

요청을 완료하기 위해 클라이언트에서 추가 조치가 필요한 경우이다. 웹 브라우저는 3xx 응답 결과에 Location 헤더가 있는 경우 Location 위치로 자동 이동(Redirect)한다.
- `Redirection은 다른 주소(URL)로 다시 요청을 보내는 것을 뜻한다. A라는 사이트에 접속했는데, 갑자기 B라는 사이트로 이동되는 경우가 '리디렉트'에 해당한다.`
- `PRG 패턴(Post-Redirect-Get): Post 요청 후 302 Found 응답으로, Get 리디렉션 요청을 유도한다. 새로고침으로 인해 중복 요청이 가는 오류를 막는 패턴이다.`

## HTTP Header와 Body
**HTTP Header의 의미**<br>
웹 브라우저와 서버 간의 통신 방법인 HTTP 통신 메세지에서 메타 데이터를 담고 있는 부분이다.
대소문자를 구분하지 않는 이름과 콜론(:) 다음에 오는 값으로 이루어져 있다. 값 앞에 붙은 빈 문자열은 무시된다.

**HTTP Body의 의미**<br>
전송하려는 실제 데이터를 의미한다. HTTP 요청에서 서버로 전달하는 JSON 데이터나 폼 데이터가 이에 해당된다. 일반적으로 헤더에 비해 복잡하고 용량이 큰 데이터를 포함한다.

### Header의 역할
클라이언트와 서버가 요청 또는 응답으로 부가적인 정보를 전송할 수 있도록 해준다. 클라이언트와 서버가 무엇을 하는지 결정하기 위해 사용된다.

#### 헤더의 종류
1. 일반 헤더(General Header)
    - 일반 헤더는 클라이언트와 서버 양쪽 모두가 사용한다. 이들은 클라이언트, 서버, 그리고 어딘가에 메세지를 보내는 다른 애플리케이션을 위해 다양한 목적으로 사용된다. 예를 들어 Date 헤더는 서버와 클라이언트를 가리지 않고 메시지가 만들어진 일시를 지칭하기 위해 사용되는 일반 목적 헤더이다. 메시지에 대한 아주 기본적인 정보를 제공한다.

    Date: Tue, 3 Oct 1974 02:16:00 GMT

2. 요청 헤더(Request Header)
    - 요청 메시지를 위한 헤더이다. 서버에게 클라이언트가 받고자 하는 데이터의 타입이 무엇인지와 같은 부가 정보를 제공한다. 예를 들어, 다음의 Accept 헤더는 서버에게 클라이언트가 자신의 요청에 대응하는 어떤 미디어 타입도 받아들일 것임을 의미한다.

    Accept: \*/\*

3. 응답 헤더(Response Header)
    - 응답 메시지는 클라이언트에게 정보를 제공하기 위한 자신만의 헤더를 갖고 있다. 예를 들어, 다음의 Server 헤더는 클라이언트에게 그가 Tiki-Hut 서버 1.0버전과 대화하고 있음을 말해준다.

    Server: Tiki-Hut/1.0

4. 엔티티 헤더(Entity Header)
    - 엔티티 본문에 대한 헤더를 말한다. 예를 들어, 엔티티 헤더는 엔티티 본문에 들어있는 데이터의 타입이 무엇인지 말해줄 수 있다. 예를 들어, 다음의 Content-Type 헤더는 애플리케이션에게 데이터가 iso-latin-1 문자집합으로 된 HTML 문서임을 알려준다.

    Content-Type: text/html; charset=iso-latin-1


5. 확장 헤더(Extension Headers)
    - 애플리케이션 개발자들에 의해 만들어졌지만 아직 승인된 HTTP 명세에는 추가되지 않은 비표준 헤더이다.


**HTTP Body의 의미**<br>
실제로 전송할 데이터이다. HTML, 이미지, 영상, JSON 등 byte로 표현할 수 있는 모든 데이터를 전송할 수 있다.

### Body의 역할
Body는 본질적인 데이터를 전달하는 역할을 한다.

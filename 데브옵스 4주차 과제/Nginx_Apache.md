# Nginx , Apache

# 1. 웹 서버의 의미

**웹 서버란?**<br>
웹 페이지와 웹 애플리케이션을 클라이언트에게 제공하는 역할을 수행하는 컴퓨터 프로그램 또는 하드웨어이다. 웹 서버는 클라이언트(웹 브라우저)로부터 요청을 받아 해당하는 웹 페이지의 내용, 이미지, 스타일 시트, 자바스크립트 파일 등을 전송하여 웹 페이지를 구성한다. 사용자가 웹 브라우저를 통해 웹 사이트에 접속하면 웹 서버는 해당 웹 페이지를 찾아서 클라이언트에게 응답하게 된다.

**웹 서버를 이용하는 이유**<br>
1. WAS 부담을 줄여주기 위해서이다.
    - WAS는 로그인, 회원가입, 개인정보 수정 등 동적 작업을 처리하는 것만으로도 작업량이 많다. 이에 정적인 파일을 클라이언트에게 전달하는 역할을 웹 서버에게 위임함으로써 WAS 작업 부담을 줄일 수 있다.
2. 보안 기능을 제공한다.
    - 웹 서버는 보안 기능을 제공하여 웹 페이지에 대한 접근을 제어할 수 있다.
3. 높은 성능을 제공한다.
    - 웹 서버는 대부분 비동기 처리 방식을 사용하여 높은 성능을 제공한다.

# 2. 웹 서버의 역할 및 작동 과정

***웹 서버의 역할**
1. 웹 리소스 제공
    - 웹 서버는 HTML 문서, 이미지, CSS 파일, JavaScript 파일 등의 웹 리소스를 제공하여 웹 페이지를 구성한다.
2. 요청과 응답 처리
    - 클라이언트로부터 오는 HTTP 요청을 받아들이고, 요청에 따른 적절한 응답을 생성하여 클라이언트에게 전송한다.
3. 웹 애플리케이션 호스팅
    - 동적인 웹 애플리케이션을 호스팅하여 서비스를 제공하는 역할을 수행한다. 웹 애플리케이션은 웹 서버에서 동작하는 프로그램으로, 데이터베이스오 상호작용하여 웹 페이지를 동적으로 생성하고 사용자의 요청에 대응한다.
4. 보안 기능
    - 웹 서버는 보안 기능을 제공하여 불법적인 접근과 공격을 방지하고 웹 사이트의 안전성을 유지한다.
5. 로깅과 모니터링
    - 웹 서버는 클라이언트의 접속 이력을 로그로 기록하고, 서버의 상태와 성능을 모니터링하여 관리자가 웹 사이트의 동작을 파악하고 최적화할 수 있도록 한다.

**웹 서버의 동작 원리**
1. 클라이언트 요청 수신
    - 사용자가 웹 브라우저를 통해 웹 사이트에 접속하면, 웹 브라우저는 해당 웹 페이지의 주소(URI 또는 URL)를 웹 서버에 요청한다.
2. 요청 분석
    - 웹 서버는 클라이언트의 요청을 받아들이고, 요청된 웹 리소스(HTML 파일, 이미지, CSS, JavaScript 등)의 위치를 파악한다.
3. 웹 리소스 검색
    - 서버는 요청된 웹 리소스가 저장된 위치를 찾아 해당 파일을 가져온다.
4. HTTP 응답 생성
    - 웹 서버는 요청에 따른 적절한 HTTP 응답을 생성한다. 이 응답에는 상태 코드, 헤더, 본문 등이 포함된다.
5. 응답 전송
    - 생성된 HTTP 응답은 다시 클라이언트로 전송된다. 이때, 웹 브라우저는 받은 응답을 해석하고 웹 페이지를 구성하는 데 필요한 리소스를 추출한다.
6. 웹 페이지 렌더링
    - 웹 브라우저는 받은 리소스를 기반으로 HTML 문서를 파싱하고, CSS 스타일링을 적용하며, JavaScript 코드를 실행하여 웹 페이지를 렌더링한다.
7. 추가 리소스 요청
    - 웹 페이지에 포함된 다른 리소스(이미지, 스크립트 파일 등)가 있다면 웹 브라우저는 해당 리소스를 서버로부터 추가로 요청하게 된다.
8. 웹 페이지 표시
    - 웹 브라우저는 받은 리소스를 모두 조합하여 최종적으로 웹 페이지를 화면에 표시한다. 사용자는 이렇게 표시된 웹 페이지와 상호작용하며 웹 사이트를 사용한다.

    웹 서버는 여러 클라이언트들로부터 동시에 요청을 받아 처리하는데, 이를 위해 병렬 처리와 동시성 관리 기술이 사용된다.

**포트 번호를 사용하는 이유**
1. 다중화
    - 하나의 컴퓨터 내에서 수많은 서비스와 애플리케이션이 동시에 실행되고 있다. 이들 모두 네티워크 통신을 필요로 할 수 있다. 포트를 사용함으로써, 하나의 아이피 주소를 가진 기기에서 동시에 여러 개의 서비스나 애플리케이션과의 통신이 가능해진다.
2. 정확한 데이터 전달
    - 포트 번호를 사용하면 수신된 데이터가 어느 서비스나 프로세스로 전달되어야 하는지 정확하게 알 수 있다.
3. 보안
    - 특정 포트를 열거나 닫음으로써 보안 정책을 적용할 수 있다. 
**자주 사용하는 포트 번호**
포트번호 | 프로토콜
:--: | :--: 
HTTP | 80
HTTPS | 443
FTP | 21

# 3. Apache 뜻

**Apache**<br>
아파치(Apache)는 가장 널리 사용되는 오픈 소스 웹 서버 소프트웨어이다. 크로스 플랫폼 지원을 제공하며, PHP, Perl, Python 등 다양한 서버 사이드스크립팅 언어와 호환된다.

**Apache의 장점**
1. 플랫폼 호환성
    - Apache는 Linux, Unix, Windows 및 MacOS를 포함한 다양한 종류의 플랫폼과 호환된다. 유연성과 사용자 정의 기능성을 우선시하는 접근 방식인 모듈식 아키텍처를 통해 새로운 기능을 빠르게 추가할 수 있다.
2. 보안
    - 조직은 암호화 및 인증 설정과 방화벽 사용 방식을 조정하여 Apache를 구성할 수 있으며, 이러한 접근 방식을 Apache구성이라고 한다.
        - 암호화: Apache서버는 컴퓨터가 인터넷을 통해 통신하는 방식을 제어하는 암호화 프로토콜 집합인 SSL/TLS(Secure Sockets Layer/Transport Layer Security)암호화 프로토콜을 지원한다.
        - 인증: Apache는 사용자가 리소스를 모니터링하고 제한할 수 있는 광범위한 강력한 인증 및 권한 부여 도구 세트를 지원한다.
        - 방호벽: Apache는 특정 컴퓨팅 환경의 요구 사항에 따라 다양한 종류의 방화벽을 사용하도록 구성할 수 있다.
3. 커뮤니티 지원
    - 크고 활발한 커뮤니티는 풍부한 자원, 튜토리얼, 지원을 제공한다
4. 트래픽 분산
    - Apache 서버의 로드 밸런싱 기능을 사용하면 인기 있는 웹사이트와 애플리케이션의 트래픽이 많은 상황에서 리소스를 최적화할 수 있다. 클라이언트 요청을 여러 서버에 분산할 수 있으므로 단일 서버가 너무 많은 요청을 수신하여 기능 및 성능을 저하시키지 않도록 한다.
    - Apache 서버는 또한 자주 요청되는 페이지를 캐싱, 저장 및 재사용하여 서버의 워크로드를 줄이고 대역폭을 효과적으로 관리하는 데 효과적이다. Apache는 다른 서버와 클라이언트 요청 사이의 중개자 역할을 하는 서버인 리버스 프록시 역할도 하여 성능과 리소스 분배를 최적화하는 데 도움이 된다.
5. 무료 오픈소스


**Apache의 단점**
1. 메모리 부하
    - 매 요청마다 새로운 프로세스를 생성하므로, 메모리 사용량의 증가로 인해 새로운 연결을 처리하지 못하는 문제가 발생한다.




# 4. Apache 모듈
**Apache 모듈**<br>
모듈은 HTTP Server의 특성을 확장하기 위해 동적으로 링크되고 로드될 수 있는 서비스 프로그램이다.

이러한 방식으로 Apache모듈은 웹 서버의 기능을 확장하는 방법을 제공한다. 선택적 모듈에서 일반적으로 추가되는 기능은 다음과 같다.
 - 보안
 - 암호화
 - 애플리케이션 지원
 - 기록
 - 다른 컨텐츠 유형에 대한 지원

# 6.Nginx 뜻

**Nginx**<br>
Nginx는 오픈소스 소프트웨어 웹서버로 시작되었다. 지금은 웹서버의 역할뿐만 아니라 프록시 서버, 캐싱, 분산 처리, 미디어 스트리밍 등을 위한 다양한 기능을 제공한다.

- 프록시: 보안상의 문제로 직접 통신을 주고받을 수 없는 두 PC사이에서 통신을 할 때 직접하지 않고 중간에서 대리로 중계를 하는 것이고, 이렇게 중계 기능을 하는 것을 프록스 서버라고 한다.
기능 뿐만 아니라 성능면에서 기존의 웹서버들보다 뛰어나 현재 Nginx는 가장 널리 사용되고 있는 웹서버이다.

**특징**
1. 비동기 이벤트 처리
    - 비동기 이벤트 기반 모델을 사용한다. 이는 동시에 많은 연결을 효율적으로 처리할 수 있도록 해주며, 이로 인해 높은 동시성과 성능을 제공한다.
2. 높은 성능과 안정성
    - 높은 성능을 유지하면서도 안정적인 서비스를 제공한다. 웹 사이트의 로딩 시간을 단축시키고 사용자 경험을 개선하는 데 도움이 된다.
3. 다양한 기능을 제공
    - 단순한 웹 서버 기능뿐만 아니라 리버스 프록시, 로드 밸런서, HTTP 캐시 등 다양한 기능을 제공한다. 이를 통해 웹 애플리케이션의 성능을 최적화하고 보안을 강화할 수 있다.
    - `로드 밸런서: 클라이언트와 서버 그룹 사이에 위치해 서버에 가해자는 트래픽을 여러 대의 서버에 고르게 분배하여 특정 서버의 부하를 덜어준다. 서버가 하나인데 많은 트래픽이 몰릴 경우 부하를 감당하지 못하고 서버가 다운되어 서비스가 작동을 멈출 수 있다. 이런 문제를 해결하기 위해 Scale up(스케일업)과 Scale out(스케일아웃) 방식 중 한 가지를 사용해 해결한다.`
4. 낮은 메모리 사용량
    - 비동기 처리 방식 덕분에, 상대적으로 낮은 메모리 사용량으로 많은 수의 동시 연결을 처리할 수 있다.
5. 유연한 설정
    - 매우 유연한 설정이 가능하여, 다양한 용도와 환경에 맞게 세부적인 조정이 가능하다.
6. 활발한 커뮤니티와 문서
    - 오픈 소스 프로젝트로, 활발한 커뮤니티와 풍부한 문서를 보유하고 있어 사용자가 문제를 해결하거나 기능을 확장하는 데 도움을 받을 수 있다.

**장점**
1. 높은 동시성 처리 능력
    - 비동기 이벤트 기반의 아키텍처를 가지고 있어 수천에서 수만 개의 동시 연결을 효육적으로 처리할 수 있다.
    - `비동기 이벤트 처리: 이벤트가 발생해도 리스너가 즉시 실행되지 않고 이벤트 큐에 등록된 후 나중에 호출된다. 리스너 실행 시점은 현재 실행중인 코드가 모두 끝난 다음이다.`
2. 낮은 메모리 사용량
    - 매우 효율적인 메모리 관리로 인해 상대적으로 낮은 자원으로도 높은 성능을 유지할 수 있다.
3. 다양한 기능 제공
    - 리버스 프록시, 로드 밸런서, HTTP 캐싱, SSL/TLS 처리 등 다양한 기능을 하나의 소프트웨어 내에서 제공한다.
4. 유연한 구성 가능
    - 강력하고 유연한 구성 시스템을 통해 다양한 사용 사례에 맞게 세밀한 설정이 가능하다.
5. 확장성
    - 모듈식 아키텍처 덕분에 사용자가 필요애 따라 기능을 추가하거나 수정할 수 있다.

**단점**
1. 동적 콘텐츠 처리
    - 동적 콘텐츠를 처리하는 데 있어 Apache와 같은 전통적인 웹 서버보다 비효율적일 수 있다.
2. 구성 파일 복잡도
    - 구성 파일은 매우 유연하지만 초보자에게는 복잡하고 어렵게 느껴질 수 있다.
3. 필요한 모듈을 컴파일 시점에 추가해야한다. 이는 Apache와 같은 서버에 비해 모듈 추가가 더 번거롭다.

# 7. Nginx 와 Apache 차이점

 Apache와 Nginx의 가장 큰 차이점은 아키텍처에 있다. Apache 서버는 각 클라이언트 요청에 대해 스레드라는 새 프로세스를 생성하는 프로세스 기반 아키텍처를 사용하지만 Ningx는 그 대신 이벤트 기반 아키텍처를 사용하여 동일한 스레드에서 많은 요청을 동시에 처리할 수 있다.

 또한 Ningx는 보다 효율적인 캐싱 기능으로 인해 Apache보다 약간 빠르게 정적 콘텐츠를 제공할 수 있다.

Nginx는 타사 플러그인 과정으로 선택되고 컴파일되기 때문에 동적으로 모듈을 로드할 수 없지만 Apache는 동적으로 로드 가능한 다양한 60개의 공식 모듈을 제공해준다.


# 8. Nginx의 주요 설정 파일 경로 및 디렉토리 역할

경로 | 설명
:--: | :--:
/etc/nginx/nginx.conf | Nginx 전역 설정 파일로, 전체 서버 동작 방식을 정의한다.
/etc/nginx/sites-available/ | 개별 사이트(서버 블록)의 설정 파일을 저장하는 디렉토리이다. 도메인이나 서비스별로 설정 파일을 만들어 효율적으로 관리할 수 있다.
/etc/nginx/sites-enabled |실제로 활성화되어 적용되는 사이트(서버 블록) 설정의 심볼릭 링크가 위치하는 디렉토리이다. 일반적으로 `sites-available` 에 있는 설정 파일을 연결하여 사용한다.
/etc/nginx/conf.d/ | 추가 설정을 저장하는 디렉토리이다. 이곳에 `.conf` 확장자 파일을 두면, Nginx가 자동으로 불러와 설정을 적용한다.
/var/www/ | 웹 서버의 기본 루트 디렉토리이다. 웹사이트의 실제 파일(HTML, CSS 등)이 이곳에 위치한다.
/var/log/nginx/ | Nginx 로그 파일이 저장되는 디렉토리이다. `access.log`와 `error.log`를 통해 접속 기록 및 오류 정보를 확인할 수 있다.

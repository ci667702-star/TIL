# AWS 기초 이론

# 1. 클라우드 의미
데이터를 보관, 정리, 분석하고 새로운 서비스를 빠르게 추진할 수 있도록 각종 인프라와 IT 기술을 빌려주는 서비스<br><br>
**IaaS(Infra as a Service)**<br>
IaaS는 인터넷을 통해 컴퓨팅, 스토리지, 네트워크 리소스와 같은 IT 인프라를 사용한 만큼만 지불하는 비즈니스 모델이다. IaaS를 사용하여 애플리케이션 및 IT 시스템을 실행하는 데 필요한 리소스를 요청하고 구성할 수 있다. 비용 효율적인 방식으로 IT 리소스에 대한 유연성과 제어를 제공한다.

*장점*
1. 속도
    - 몇 분 안에 원하는 수의 리소스를 프로비저닝하고 새로운 아이디어를 훨씬 빠르게 테스트하고 출시할 수 있다. 다른 사람들이 IT 인프라와 컴퓨팅 리소스를 완전히 관리하기 때문에 핵심 비지니스 활동에 집중할 수 있다.
2. 성능
    - 지리적으로 분산된 데이터 센터를 사용하여 고객과 물리적으로 가까운 위치에서 애플리케이션을 스케일 업 할 수 있다.
3. 신뢰성
    - IaaS 공급 업체는 교체용 가상 머신을 빠르게 예측 가능하게 실행할 수 있는 매우 안정적인 환경을 제공한다.
4. 백업 및 복구
    - IaaS 공급 업체에는 백업 및 재해 복구용 무제한 인프라에 대한 액세스를 제공한다.
<br>
**PaaS(Platform as a Service)**<br>
일반적으로 개발자가 인터넷을 통해 액세스할 수 있는 안전한 온라인 플랫폼으로 제공되므로 어디서나 프로젝트를 작업하고 다른 팀원과 자유롭게 공동작업할 수 있다. 애플리케이션을 PaaS시스템에서 직접 빌드하여 완성 즉시 배포할 수 있다.

*장점*
1. TTM(time to market) 단축
    - 복잡한 리프트 절차가 필요하지 않다. 개발자가 빌드하거나 관리할 필요가 없는 완벽한 애플리케이션 개발 플랫폼에 즉시 액세스할 수 있으므로 개발 및 배포 시간이 확보된다.
2. 유지보수 수요 감소
3. 비용 효율적인 가격 책정
4. 손쉬운 확장성
5. 유연한 액세스
6. 공유 보안
<br>


**SaaS(Software as a Service)**<br>
SaaS는 사용자가 소프트웨어를 직접 구매하여 설치하는 방식이 아닌 인터넷을 통해 원격으로 접속하여 사용하는 소프트웨어 제공 방식이다 SaaS모델에서는 고객이 소프트웨어를 자신의 컴퓨터나 서버에 설치하거나 유지 보수하지 않고 인터넷을 통해 소프트웨어 애플리케이션에 접근할 수 있다.

*특징*
1. 클라우드 기반 접근
    - SaaS 애플리케이션은 클라우드에서 호스팅되어 웹 브라우저를 통해 접근 가능
2. 구독 기반 모델
3. 중앙 관리

*장점*
1. 편리하고 쉬운 접근
    - 인터넷만 연결되어 있다면 언제 어디서나 접근이 가능한 편리한 사용 방식을 제공한다.
2. 비용 효율성 극대화
3. 간편한 설치 및 유지 보수
    - 클라우드 기반으로 제공되기 때문에 소프트웨어의 설치 및 유지보수가 간편하다.
4. 필요에 다라 확장이 용이
5. 보안
    - 데이터 보안과 백업 및 복원과 같은 보안 문제를 처리하므로 고객은 보안 문제에 대해 걱정할 필요가 없다.
<br>

<br>

# 2. AWS 의미
**AWS란?**<br>
AWS는 클라우드 컴퓨팅을 통해 전 세계적으로 연결된 원격 컴퓨터를 활용하여 다양한 IT 리소스를 제공하는 주문형 서비스이다.<br><br>
**특징**
1. 정교하고 확장성 높은 애플리케이션 구현
    - AWS의 다양한 IT 리소스를 활용하여 복잡한 애플리케이션을 구현할 수 있다.
    - 대규모 API 요청을 안정적으로 처리할 수 있는 인프라를 제공한다.
2. 글로벌 보안 표준 준수
    - ISO 27001, HIPAA, PCI DSS 등 국제적으로 인정받는 보안 표준을 준수한다.
    - 데이터 암호화, 접근 제어 등 고급 보안 기능을 기본으로 제공한다.
3. 자동 리소스 관리
    - Auto Scaling 기능을 통해 트래픽 변동에 따라 자동으로 리소스를 조절한다.
    - 비용 최적화와 성능 유지를 동시에 달성할 수 있다.
4. 신뢰성과 가용성
    - 다중 가용영역 아키텍처를 통해 높은 가용성을 보장한다.
    - 데이터 복제 및 백업 기능으로 데이터 손실 위험을 최소화한다.

<br>

**AWS의 장점**
1. 유연성과 확장성
    - 사용자들이 필요한 리소스를 필요한 만큼 늘리거나 줄일 수 있도록 유연성과 확장성을 제공한다. 이를 통해 사용자는 트래픽이나 수요 증가에 대응할 수 있으며, 필요하지 않은 리소스는 낭비하지 않을 수 있다.
2. 보안
    - AWS는 전세계적인 데이터 센터 네트워크를 통해 데이터를 보호하며, 보안 인증 및 규정 준수, 보안 이벤트 알림 등 다양한 보안 서비스를 제공하고 있다.
3. 비용 효율성
    - 기업들이 필요한 만큼의 리소스만 지불하도록 하여 낭비를 최소화하고 인프라를 구축하고 운영하는 데 필요한 비용을 절감할 수 있다.
4. 글로벌 인프라스트럭처
    - AWS는 전 세계에 분산된 데이터 센터와 콘텐츠 전달 네트워크(CDN)를 보유하며, 글로벌 사용자에게 저지연 시간 및 빠른 속도로 서비스를 제공할 수 있다.
<br>

**AWS의 단점**
1. 복잡성
    - AWS는 다양한 서비스와 옵션을 제공하며, 이로 인해 진입장벽이 높다.
2. 인터넷 연결 의존성
    - 인터넷 연결에 의존하는 서비스는 네트워크 문제에 민감할 수 있다.


# 3. AWS 인프라 환경

**AWS 글로벌 인프라**<br>
AWS의 글로벌 인프라는 전 세계에 걸쳐 확장된 클라우드 컴퓨팅 플랫폼으로, 기업들이 고성능, 안정성, 확장성을 갖춘 서비스를 제공할 수 있도록 지원한다.
<br>
<br>

*주요 구성 요소*

1. 리전(Region)<br>
    AWS의 주요 서비스가 제공되는 지리적 위치
    - 최소 2개 이상의 가용 영역으로 구성
    - 각 리전마다 사용 가능한 서비스와 리소스가 다를 수 있음
    - 데이터 주권 및 규제 준수를 위해 중요
2. 가용 역역(Availability Zone, AZ)<br>
    리전 내의 독립된 데이터 센터 그룹
    - 별도의 전력, 냉각, 물리적 보안을 갖춤
    - 초저지연 네트워크로 상호 연결
    - 고가용성, 내결함성, 확장성 제공
3. 엣지 로케이션(Edge Location)<br>
    CDN 서비스를 제공하는 캐시 서버 위치
    - 사용자와 가까운 위치에 콘텐츠 캐시 저장
    - 지연 시간 감소 및 성능 향상
    - AWS CLoudFront 같은 서비스에서 사용
    
    -  `CDN: 콘텐츠를 보다 빠르게 전송하는 기술로, 속도 개선과 회선 비용 절감에 용이하다.`
    
    - 주요 서비스
        - CloudFront(CDN):콘텐츠 전송 네트워크
        - Route 53:도메인 네임 시스템(DNS) 웹 서비스
4. 로컬 영역(Local Zones)<br>
    특정 지역에 AWS 서비스를 더 가깝게 제공하는 소규모 인프라
    - 매우 낮은 지연 시간이 필요한 애플리케이션을 위해 설계
    - 주로 대도시 지역에 위치
   


# 4. AWS **Storage**

**DAS(Direct Attached Storage)**<br>
PC나 서버에 다이렉트로 꽂아서 사용하는 스토리지로 서버와 하드웨어를 1:1로 연결한다. <br>각 서버는 자신이 직접 파일 시스템을 관리할 수 있다.
또한, 하드를 여러개 끼워 사용하는 서버의 외장하드라고 생각하면 된다.
서버 외부에서 직접 저장장치를 연결하므로 속도는 빠르고 확장은 쉽지만 연결 수에 한계가 있다.
<br>

**NAS(Network Attached Storage)**<br>
네트워크 연결 스토리지는 직원들이 네트워크를 통해 효과적으로 협업할 수 있도록 데이터를 지속적으로 사용할 수 있게 하는 파일 전용 스토리지 디바이스이다. <br>모든 컴퓨터 네트워크에는 서버에 요청을 전송하는 상호 연결된 서버 시스템과 클라이언트 시스템이 있다. NAS 디바이스는 데이터 스토리지 및 파일 공유 요청만 처리하는 특수 서버로서, 프라이빗 네트워크에 빠르게 안전하며 안정적인 스토리지 서비스를 제공한다.
<br>

*중요한 이유*<br>
효과적이고 확장 가능하며 저렴한 스토리지를 제공하기 때문이다. <br>다른 서버에 비해 NAS 파일 서버는 더 빠른 데이터 액세스를 제공하고 구성 및 관리가 더 용이하다. 또한 프라이빗 이메일 시스템, 회계 데이터베이스, 급여, 비디오 녹화 및 편집, 데이터 로깅, 비지니스 분석을 포함한 다양한 비지니스 애플리케이션을 지원할 수 있다.

*이점*
1. 조직을 위한 프라이빗 클라우드 배포
    - 프라이빗 클라우드는 조직의 자체 데이터 센터에서 리소스를 호스팅하는 클라우드로서, 내부 하드웨어 리소스 또는 서드 파티에서 제공하는 별도의 인프라를 기반으로 할 수 있다. NAS 어플라이언스를 사용하여 조직에 프라이빗 클라우드 스토리지를 배포할 수 있다.
2. 소규모 기업을 위한 유연한 로컬 스토리지 솔루션
    - 조직의 규모와 요구사항에 따라 NAS 시스템을 사용자 지정할 수 있다.

*구조*
- 물리적 스토리지 드라이브
    - NAS 디바이스는 2~5개의 하드 드랄이브를 포함할 수 있어 대량의 스토리지 용량을 제공한다. 여러 물리적 드라이브는 논리적으로 중복 스토리지 컨테이너(RAID)로 정렬된다.
- 중앙 처리 장치(CPU)
    - NAS 디바이스에는 컴퓨팅 인텔리전스와 파일 시스템 관리 기능을 제공하는 CPU가 있다. CPU는 파일을 처리 및 제공하고, 여러 사용자를 관리하고, 원하는 경우 클라우드와 통합하기 위해 데이터를 읽고 쓴다.
- 운영 체제
    - 운영 체제는 스토리지 디바이스 하드웨어와 해당 사용자 사이의 소프트웨어 인터페이스이다. 복잡한 네트워크 연결 저장 장치 디바이스는 자체 운영 체제와 함께 제공되지만 일부 더 단순한 디바이스에는 운영 체제가 없을 수 있다.
- 네트워킹 인터페이스
    - NAS 장치는 네트웤이 인터페이스를 사용하여 네트워크에 연결한다. 네트워크 연결은 이더넷 케이블 또는 Wi-Fi일 수 있다.

**SAN(Storage Area Network)**<br>
서버와 저장장치를 NAS와는 다르게 Fiber channel(광채널) switch로 연결한 고속데이텉 네트워크이다.
<br>
저장장치를 향상시켜 장치가 롴러 연결장치로 서버의 운영 체제에 표시되도록 한다.
<br>
DAS와 NAS의 단점을 보완하여 서버당 접속 스토리지 수 제한 해결하였고 전용 파이버 채널 스위치(광채널)를 둠으로써 빠른 속도의 연결을 유지하는 것이 특징이다. 또한, NAS와 달리 파일이 아닌 블록 단위의 I/O가 기본 단위이다.
<br>

**스냅샷**<br>
EBS를 저장하는 효율적인 방법으로서, 특정 시간에 EBS 볼륨 상태의 저장본을 뜻한다. 필요시 스냅샷을 통해 특정시간의 저장 데이터에 대한 EBS 복구가 가능하다.

*특징*
1. 시점 기반 백업
    - 스냅샷 생성 시점의 데이터 상태를 정확히 보존한다.
    - 증분 백업 방식으로 효율적으로 저장공간을 사용한다.
    - 빠른 백업 생성과 복구를 지원한다.
2. 독립적 관리
    - 원본 블록 스토리지와 독립적으로 관리된다.
    - 원본이 삭제되어도 스냅샷은 보존된다.
    - 여러 스냅샷 버전을 동시에 유지 가능하다.
3. 유연한 활용
    - 새로운 블록 스토리지 생성의 기반으로 사용된다.
    - 다른 가상머신에 복제하여 활용된다.
    - 개발/테스트 환경 구성에 활용된다.
<br><br>

# 5. AWS 네트워크
**VPC**<br>
AWS 클라우드에서 다른 가상 네트워크와 논리적으로 격리되어 있는 AWS계정 전용 가상 네트워크이다. 필요 시, 개발, 프로덕션, 테스트 등과 같이 각각의 워크 로드를 다른 VPC 상에 배치하여 논리적으로 격리할 수 있다.
<br>

*특징*
- IP 주소, 서브넷, 라우팅 테이블 등을 정의하여 AWS 자원들이 안전하게 통신할 수 있다.
- VPC는 하나의 리전에 포함되며 리전 안에 있는 모든 가용 영역의 리소스를 호스팅 할 수 있다.
- 각 VPC는 완전히 독립적이다. 기본적으로 VPC간의 통신은 제한된다. 필요 시, VPC Peering 서비스를 사용해 두 VPC 사이의 네트워크를 연결하여 서로 통신할 수 있다.
- VPC 사용자가 지정하는 pricate IP 주소의 범위를 예약한다.
<br><br>

**Subnet**<br>
서브넷은 리소스 그룹을 격리할 수 있는 VPC의 IP 주소 범위의 세그먼트 또는 파티션이다. 각 서브넷은 하나의 가용 영역 내에서만 존재하며, 하나의 가용 영역에 여러 개의 서브넷이 포함될 수 있다. 한 가지 유의해야 할 점은, 서브넷은 언제나 5개의 Ip 주소를 기본적으로 에약한다. 따라서 의미 없이 서브넷을 나누게 되면 낭비되는 Ip 주소가 생길 수 있다.

서브넷은 인터넷과 통신할 수 있는 지의 여부로 Public/Private Subnet으로 나뉜다.
- 퍼블릭 서브넷(public subnet)<br>
  인터넷에 대한 Inbound/Outbound 액세스를 지원하도록 "인터넷 게이트웨이"에 대한 라우팅 테이블 항목을 포함하고 있는 서브넷을 퍼블릭 서브넷이라고 한다.
    - `inbound: 데이터가 서버로 들어오는 경우<br>outbound: 데이터가 서버 밖으로 나가는 경우`
- 프라이빗 서브넷(private subnet)<br>
  "인터넷 게이트웨이"에 대한 라우팅 테이블 항목이 없고, 퍼블릭 인터넷에서 직접 액세스할 수 없는 서브넷을 프라이빗 서브넷이라고 한다. 일반적으로 인터넷과의 통신이 필요한 경우에. 제한된 outbound 퍼블릭 인터넷 액세스를 지원하기 위해 "Nat(Network address Translation)게이트웨이"를 사용한다.
<br><br>


**라우팅 테이블**<br>
라우팅 테이블은 서브넷 또는 게이트웨이의 네트워크 트래픽이 전송되는 위치를 결정하는 데 사용되는 규칙(이 규칙을 Routing이라고 한다.)의 집합이다.
<br>
각 VPC에는 "local"로 향하는 기본 라우팅 테이블이 존재한다. 이 기본 라우팅 테이블은 다른 라우팅 테이블과 명시적으로 연결되지 않은 모든 서브넷의 라우팅을 제어한다.

**인터넷 게이트웨이**<br>
VPC 내에 있는 인스턴스와 인터넷 간에 통신을 허용할 수 있게 해주는 도구로 수평 확장되고 가용성이 높다.
인터넷 게이트웨이는 인터넷으로 라우팅 가능한 트래픽에 대한 라우팅 테이블에 대상을 제공한다.
또한 인터넷 게이트웨이는 public IPv4 주소가 할당된 인스턴스에 대해 NAT(네트워크 주소 변환)를 수행한다.

**NAT 게이트웨이**<br>
네트워크 주소 변환(NAT) 서비스이다. NAT 게이트웨이를 사용하면 프라이빗 서브넷의 인스턴스가 VPC 외부의 서비스에 연결할 수 있지만 외부 서비스에서 이러한 인스턴스와의 연결을 시작할 수는 없도록 할 수 있다.

**Security Group**<br>
보안 그룹은 인스턴스 수준에서 인바운드와 아웃바운드 트래픽을 제어하는 가상 방화벽이다.
*특징*
- 인스턴스에 대한 인바운드/아웃바운드 트래픽을 제어하는 방화벽 역할을 한다.
- 인스턴스 단위로 적용된다
- 특정 그룹을 지정했을 때만 인스턴스에 적용된다.


**NACL(Network Access Control List)**<br>
서브넷 수준에서 트래픽을 제어하는 방화벽이다. 서브넷에 출입하는 모든 트래픽을 검사한다.

*특징*
- 서브넷 단위에 적용된다.
- NACL이 설정된 서브넷 안의 모든 인스턴스에 적용한다.
- 1개의 VPC에 NAXL 최대 200개 까지 생성 가능하다.
- 1개의 NACL은 여러개의 서브넷에 적용 가능하다.
- 서브넷은 하나의 NACL만 적용 가능하다.
- 규칙 번호가 낮은 것부터 우선 적용된다.

항목 |Security Group | AWS Network ACL
:---: | :---: | :---:
적용 수준 | 인스턴스 | 서브넷
상태 | Stateful(상태 저장) | Stateless(상태 비저장)
규칙 유형 | 허용 | 허용+거부
규칙 평가 | 모든 규칙 평가 | 순서대로 평가
기본 인바운드 | 모두 거부 | 기본 NACL은 모두 허용
적용 방식 | 인스턴스에 직접 연결 | 서브넷의 모든 인스턴스에 적용

- `Stateful(상태 저장): 인바운드로 들어온 트래픽의 응답이 자동으로 아웃바운드로 허용된다는 의미이다.`
- `Stateless(상태 비저장): 인바운드와 아웃바운드 규칙을 각각 별도로 설정해야 한다는 의미이다.`

# 6. AWS Architecture
**아키텍처의 의미**<br>
AWS 아키텍처는 개발자가 클라우드 애플리케이션을 설계하기 위해 사용하는 방법, 기술 및 프레임워크의 결합이다.

**3-Tier Architecture**<br>
3가지 계층으로 구분하여 시스템을 구성하는 것을 말하며, 계층간의 통신이 순차적으로 이루어진다.

1. 프레젠테이션 계층
    - 역할: 사용자와 사오작용하는 인터페이스르 제공
    - 구성 요소: 웹 브라우저, 모바일 앱, 데스크 탑 애플리케이션 등
    - 주요 기능: 데이터를 사용자에게 표시하고, 사용자의 입력을 받아 비즈니스 계층으로 전달
2. 비즈니스 계층
    - 역할: 애플리케이션의 핵심 로직을 처리
    - 구성 요소: 웹 서버, 애플리케이션 서버 등
    - 주요 기능: 프레젠테이션 계층으로부터 입력을 받아 처리/ 데이터 계층과 상호작용
3. 데이터 계층
    - 역할: 데이터를 영구적으로 저장 및 관리
    - 구성 요소: 데이터베이스 관리 시스템(DBMS), 파일 시스템 등
    - 주요 기능: 비즈니스 계층으로부터 요청받은 데이터를 처리

**2-Tier Architecutre**<br>
2티어 아키텍처는 정레젠테이션 티어와 데이터 티어로 구성된 최초의 클라이언트-서버 아키텍처로 비즈니스 로직은 프레젠테이션 티어, 데이터 티어 또는 둘 다에 존재한다. 2티어 아키텍처에서는 프레젠테이션 티어, 즉 최종 사용자가 데이터티어에 직접 액세스할 수 있으며 비즈니스 로직이 제한되는 경우가 많다.    

**설계 방법**
1. VPC 구성
    - VPC 생성
    - Subnet 분할
    - Internet Gateway & NAT Gateway
2. Bastion Host 배치
    - EC2 인스턴스 생성
    - Public Subnet에 배치
3. RDS 배치
    - Private Subnet에 배치
    - Multi-AZ 설정을 켜서 데이터 유실을 방지
4. 백엔드 서버 배치
    - Private Subnet에 배치
    - Auto Scaling Group을 설정하여 트래픽에 따라 서버 대수가 조절되게 한다.
5. ALB 배치
    Public Subnet에 배치하여 사용자 요청을 받는다.
6. 보안 그룹 수정

# 7. AWS Services

## **Computing Service**

**EC2**<br>
CPU, Memory, OS등을 가지는 컴퓨터이다. 하지만 실제 서버 컴퓨터와는 다르게 가상의 컴퓨팅 환경으로, 쉽게 컴퓨팅 파워를 조정할 수 있다. AWS 글로벌 인프라 안에서 여러 가용 영역에 걸쳐 배포하여 안정성과 고가용성을 만족할 수 있다.
- `인스턴스: `

**EBS**<br>
일관되고 지연 시간이 짧은 성능을 제공하는 영구적인 블록 스토리지 볼륨이다. 접근을 위해 컴퓨팅 파워가 필요한 HDD와 SDD와 같이, EBS는 홀로 사용될 수는 없고 EC2 인스턴스에 연결하여 사용할 수 있다.

**S3**<br>
인터넷용 Object 스토리지로 기본적으로 HTTP와 HTTPS 프로토콜의 인터넷을 통한 액세스 방식을 가진다. 웹에서 언제 어디서나 원하는 양의 데이터를 저장하고 검색할 수 있다. 고도의 확장성과 안정성, 속도 및 내구성을 가지고 있다. S3에 저장할 수 있는 객체 수에 제한이 없고, 저장할 수 있는 객체의 크기는 최대 5TB이다.

- 버킷<br>
  버킷은 S3에 저장된 객체에 대한 컨테이너이다. 버킷의 이름은 리전에 상관없이 전역적으로 고유해야한다.
- 객체(Object)<br>
  객체는 S3에 저장디는 기본 엔티티이다. 객체는 데이터와 메타 데이터로 구성된다. "데이터"는 AWS S3 console에서는 볼 수 없고, 권한을 갖춘 사용자가 HTTP/HTTPS를 통한 접근 시 볼 수 있다. "메타 데이터"는 객체를 설명하는 key-value의 집합이다. 수정한 날짜, Content-Type과 같은 HTTP 메타 정보 등을 포함한다.
- 객체 버전 관리<br>
  S3는 객체가 중복될 시 덮어쓴다. 이는 실수에 의한 데이터 손실을 유발할 수 있다.
- S3의 보안<br>
  기본적으로 버킷, 객체 및 하위 리소스는 모두 비공개이다. 즉, 리소스를 만든 계정 소유자만 액세스할 수 있다. S3는 리소스 기반 정책과 사용자 정책으로 접근에 대해 정의한다.
**EFS**<br>
파일 스토리지 서비스로 여러 EC2 인스턴스가 파일을 공유할 수 있다.
- 확장성과 가용성이 뛰어나다.

## **Networking Service**
**VPC**<br>
5번 참조

**Route 53**<br>
DNS로, 사용자의 도메인을 관리하고 트래픽을 적절한 서버로 라우팅하는 서비스이다.
- 고가용성과 확장성을 제공하며 전 세계적인 분산 DNS 인프라를 제공한다.

**ELB**
로드 밸런서로, 여러 EC2 인스턴스에 트래픽을 분산시킨다.
- 애플리케이션의 가용성을 높이고 부하에 따라 자동으로 트래픽을 분배한다.

ELB는 크게 리스너와 대상 그룹으로 구성된다.
- 리스너: 외부 요청을 받아들이는 역할을 한다. 서비스 포트를 통해 요청을 수신하며 요청이 유효하다면 대상 그룹으로 전달한다. 리스너는 포트 변환을 통해 들어온 요청을 다른 포트로 전달할 수도 있다.
- 대상 그룹: 트래픽이 분산될 리소스들의 집합이다. EC2 인스턴스, 컨테이너, IP주소 등이 포함될 수 있으며, 각 대상은 서비스 포트를 열어 둬야 한다. ELB는 대상 그룹에 등록된 리소스의 상태를 Health Check로 지속적으로 확인한다. 정상 상태인 대상에게만 요청이 전달된다.

> 과정
> 1. 사용자가 URL로 접근한다.
> 2. 요청이 ELB의 Listener로 들어온다.
> 3. Listener는 요청을 검사한 후 적절한 대상 그룹으로 전달한다.
> 4. ELB는 대상 그룹 내에서 Health Check가 정상인 리소스를 찾아 트래픽을 분배한다.

*종류*
1. ALB(Application Load Balancer)
    - L7에서 동작
    - HTTP/HTTPS 트래픽 처리에 최적화
    - 경로 기반 라우팅, 트래픽 처리에 최적화
    - 고정 IP 주소 사용 불가
2. NLB(Network Load Balancer)
    - L4에서 동작
    - TCP/UDP 트래픽에 적합
    - 초당 수백만 건의 요청을 처리할 정도로 고성능이다
3. GLB(Gateway Load Balancer)
    - L3에서 동작
    - 보안 장비(Virtual Appliance)와 함께 사용
    - 트래픽을 투명하게 전달하고 모니터링 가능
4. CLB(classic Load Balancer)
    - Elastic Load Balancing 이전 세대의 로드 밸런서이다.
    - L4와 L7에서 동작
    - HTTP/HTTPS/TCP/SSL 트래픽을 처리
    - 현재는 사용하지 않음
    


기능 | ALB | NLB | GLB
:--: | :--: | :--: | :--:
로드 밸런서 유형 | 계층 7 | 계층 4 | 계층 3 게이트웨이 + 계층 4 로드 밸런싱
대상 유형 | IP, 인스턴스, Lambda | Ip, 인스턴스, ALB | IP, 인스턴스
흐름/ 프록시 동작 종료 | 예 | 예 | 아니요
프로토콜 리스너 | HTTP, HTTPS, gRPC | TCP, UDP, TLS | IP

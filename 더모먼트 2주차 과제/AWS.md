

# 1. AWS 의미
**AWS란?**<br>
AWS는 클라우드 컴퓨팅을 통해 전 세계적으로 연결된 원격 컴퓨터를 활용하여 다양한 IT 리소스를 제공하는 주문형 서비스이다.<br><br>
**특징**
1. 정교하고 확장성 높은 애플리케이션 구현
    - AWS의 다양한 IT 리소스를 활용하여 복잡한 애플리케이션을 구현할 수 있다.
    - 대규모 API 요청을 안정적으로 처리할 수 있는 인프라를 제공한다.
2. 글로벌 보안 표준 준수
    - ISO 27001, HIPAA, PCI DSS 등 국제적으로 인정받는 보안 표준을 준수한다.
    - 데이터 암호화, 접근 제어 등 고급 보안 기능을 기본으로 제공한다.
3. 자동 리소스 관리
    - Auto Scaling 기능을 통해 트래픽 변동에 따라 자동으로 리소스를 조절한다.
    - 비용 최적화와 성능 유지를 동시에 달성할 수 있다.
4. 신뢰성과 가용성
    - 다중 가용영역 아키텍처를 통해 높은 가용성을 보장한다.
    - 데이터 복제 및 백업 기능으로 데이터 손실 위험을 최소화한다.

<br>

**AWS의 장점**
1. 유연성과 확장성
    - 사용자들이 필요한 리소스를 필요한 만큼 늘리거나 줄일 수 있도록 유연성과 확장성을 제공한다. 이를 통해 사용자는 트래픽이나 수요 증가에 대응할 수 있으며, 필요하지 않은 리소스는 낭비하지 않을 수 있다.
2. 보안
    - AWS는 전세계적인 데이터 센터 네트워크를 통해 데이터를 보호하며, 보안 인증 및 규정 준수, 보안 이벤트 알림 등 다양한 보안 서비스를 제공하고 있다.
3. 비용 효율성
    - 기업들이 필요한 만큼의 리소스만 지불하도록 하여 낭비를 최소화하고 인프라를 구축하고 운영하는 데 필요한 비용을 절감할 수 있다.
4. 글로벌 인프라스트럭처
    - AWS는 전 세계에 분산된 데이터 센터와 콘텐츠 전달 네트워크(CDN)를 보유하며, 글로벌 사용자에게 저지연 시간 및 빠른 속도로 서비스를 제공할 수 있다.
5. 고도로 가용한 서비스
    - AWS는 다양한 리전과 가용 영역을 제공하여 고가용성을 보장하고 장애 복구를 지원한다.
6. 데이터베이스 선택
    - AWS는 사용자에게 다양한 유형의 데이터베이스 선택권을 제공한다. 사용자는 최고의 성능을 자랑하고 편리한 데이터베이스를 이용할 수 있다.
<br>

**AWS의 단점**
1. 복잡성
    - AWS는 다양한 서비스와 옵션을 제공하며, 이로 인해 진입장벽이 높다.
2. 인터넷 연결 의존성
    - 인터넷 연결에 의존하는 서비스는 네트워크 문제에 민감할 수 있다.
3. 데이터 전송 비용
    - 데이터 전송은 추가 비용이 발생할 수 있어 트래픽이 많은 애플리케이션은 고려해야한다.
4. 데이터 소유권 및 보안 우려
    - 클라우드로 데이터를 이전할 때 데이터 소유권과 보안에 대한 우려가 발생할 수 있다.
5. 커스터머 지원 비용
    - AWS 프리티어 서비스 이상을 사용할 경우, 커스터머 지원 비용이 발생할 수 있다.


# 2. AWS 인프라 환경

**AWS 글로벌 인프라**<br>
AWS의 글로벌 인프라는 전 세계에 걸쳐 확장된 클라우드 컴퓨팅 플랫폼으로, 기업들이 고성능, 안정성, 확장성을 갖춘 서비스를 제공할 수 있도록 지원한다.
<br>
<br>

*주요 구성 요소*

1. 리전(Region)<br>
    - 리전을 구축하여 해당 위치에서 가장 가까운 곳에서 클라우드 서비스를 이용할 수 있도록 서비스를 제공한다. 
    - 각 리전은 격리되도록 설계되었다. 이를 통해 가장 강력한 내결함성 및 안정성을 달성할 수 있다.

2. 가용 역역(Availability Zone, AZ)<br>
    - 우리가 흔히 알고 있는 데이터센터를 말한다.
    - AWS 하나의 리전에 다수의 가용 영역을 보유하고 있고, 가용 영역이 위치한 데이터 센터는 같은 리전이라도 지리적으로 멀리 떨어져있다. 가용 영역이 이렇게 물리적으로 떨어져 있는 이유는, 하나의 가용 영역이 재해, 정전, 테러 화재 등 다양한 이유로 작동 불능이 되더라도 다른 가용영역에서 서비스를 재개할 수 있도록 하기 위함이다.
3. 엣지 로케이션(Edge Location)<br>
    - AWS의 CDN 제품인 CloudFront의 캐싱 콘텐츠가 위치하는 곳이다. CDN은 인터넷 상에 콘텐츠를 캐싱하여 좀 더 빠르게 사용자에게 전달하기 위한 기술이다.
    
    - CDN: 콘텐츠를 보다 빠르게 전송하는 기술로, 속도 개선과 회선 비용 절감에 용이하다.
    - 주요 서비스
        - CloudFront(Contents Delivery Network): 개발자 친화적 환경에서 짧은 지연 시간과 빠른 전송 속도로 데이터, 동영상, 애플리케이션 및 API를 전세계 고객에서 안전하게 전송하는 고속 콘텐츠 네트워크 서비스이다.
        - Route 53:DNS로 사용자의 도메인을 관리하고 트래픽ㅇ르 가장 적합한 서버로 라우팅하는 서비스이다.
4. 로컬 영역(Local Zones)<br>
    - AWS 로컬 영역은 컴퓨팅, 스토리지, 데이터베이스 및 기타 일부 AWS 리소스를 대규모 인구 및 산업 센터와 가까운 곳에 배치한다. 로컬 영역을 사용하여 사용자에게 애플리케이션에 대해 짧은 지연 시간 액세스를 제공할 수 있다.

# 3. AWS **Storage**

**DAS(Direct Attached Storage)**<br>
PC나 서버에 다이렉트로 꽂아서 사용하는 스토리지로 서버와 하드웨어를 1:1로 연결한다. <br>각 서버는 자신이 직접 파일 시스템을 관리할 수 있다. 가장 전통적인 방식의 스토리지로, 별도의 네트워크를 거치지 않기 때문에 설치가 간편하고 성능도 우수하다. 하지만 확장성과 공유 측면에서는 한계가 있다.
<br>

*장점*
1. 설치와 운용이 쉽다.
2. 직접 연결하기 때문에 속도가 다른 스토리지에 비해서 빠르다.
3. 가격 측면에서도 다른 장비에 비해 저렴하다.

*단점*
1. 서버 포트수에 한계가 있다.
2. 서버가 다운될 경우에는 스토리지 사용이 불가능하다.
3. 스토리지 장치 해제 시 저장된 데이터 유실 가능성이 존재한다.

**NAS(Network Attached Storage)**<br>
네트워크 연결 스토리지는 직원들이 네트워크를 통해 효과적으로 협업할 수 있도록 데이터를 지속적으로 사용할 수 있게 하는 파일 전용 스토리지 디바이스이다. <br>모든 컴퓨터 네트워크에는 서버에 요청을 전송하는 상호 연결된 서버 시스템과 클라이언트 시스템이 있다. NAS 디바이스는 데이터 스토리지 및 파일 공유 요청만 처리하는 특수 서버로서, 프라이빗 네트워크에 빠르게 안전하며 안정적인 스토리지 서비스를 제공한다. 일반적인 파일 공유와 백업 용도로 많이 사용된다.
<br>

*장점*
1. 데이터를 중앙 집중화하기 좋아 접근하기 쉽다
2. 백업과 설치가 간단하다.

*단점*
1. 사용자가 증가하면 속도가 느려진다.
2. 높은 보안 수준을 요구하는 곳에서 사용이 적합하지 않다.

**SAN(Storage Area Network)**<br>
서버와 저장장치를 NAS와는 다르게 Fiber channel(광채널) switch로 연결한 고속데이터 네트워크이다. 서로 다른 종류의 데이터 저장 장치를 한 데이터 서버에 연결하여 총괄적으로 관리해주는 네트워크를 의미한다.
<br>
저장장치를 향상시켜 장치가 롴러 연결장치로 서버의 운영 체제에 표시되도록 한다.
<br>
DAS와 NAS의 단점을 보완하여 서버당 접속 스토리지 수 제한 해결하였고 전용 파이버 채널 스위치(광채널)를 둠으로써 빠른 속도의 연결을 유지하는 것이 특징이다. 또한, NAS와 달리 파일이 아닌 블록 단위의 I/O가 기본 단위이다.
<br>

### 클라우드 스토리지의 세가지 유형
1. Block Storage(블록 스토리지)
    - 데이터를 고정된 블록 단위로 저장하며, 빠르고 짧은 지연 시간으로 고성능을 제공한다. OS에서 로컬 디스크처러 사용할 수 있기에 빠른 데이터 액세스가 필요한 애플리케이션에 적합하다.
2. FIle Storage(파일 스토리지)
    - 파일과 데릭터리의 계층 구조를 통해 데이터를 저장하는 방식이다. 동시에 복수의 서버나 사용자가 하나의 파일 시스템을 공유할 수 있다.
3. Object Storage(오브젝트 스토리지)
    - 대용량의 비정형 데이터를 저장하기에 최적화된 방식이다. 파일 자체뿐 아니라 메타데이터와 함께 하나의 객체로 저장되며, 주로 정적 웹 컨텐츠를 호스팅하거나 여러 데이터센터에 데이터를 분산시키는 경우에 적합하다.

스토리지 분류 | AWS 스토리지 서비스
:--|:--
Block Storage | Amazon EBS
File Storage | Amazon EFS
Object Storage | Amazon S3

# 4. AWS 네트워크
**VPC**<br>
AWS 계정 사용자 전용 가상 네트워크이다. 사용자는 자기가 원하는 대로 IP 주소 범위 선택, 서브넷 생성, 라우팅 테이블 및 네트워크 게이트웨이 구성 등 가상 네트워크 환경을 구성해 VPC를 생성할 수 있다.
한마디로 AWS용 나만의 개인 네트워크 망 데이터센터라도 이해하면 된다.


<br>

*특징*
*장점*
1. 보안 강화
    - 연결을 보호 및 모니터링하고, 트래픽을 차단하며, 가상 네트워크 내부의 인스턴스 액세스를 제한한다.
2. 시간 절약
    - 가상 네트워크를 설정, 관리 및 검증하는 데 소요되는 시간을 줄인다.
3. 환경 관리 및 제어
    - 자체 IP 주소 범위를 선택하고, 서브넷을 생성한 후 라우팅 테이블을 구성하여 가상 네트워크를 사용자 지정할 수 있다.
<br><br>

**Subnet**<br>
서브넷은 VPC의 Ip 주소 범위이다.

서브넷은 인터넷과 통신할 수 있는 지의 여부로 Public/Private Subnet으로 나뉜다.
- 퍼블릭 서브넷(public subnet)<br>
  서브넷에 인터넷 게이트웨이로 직접 연결되는 경로가 있다. 퍼블릭 서브넷의 리소스는 퍼블릭 인터넷에 액세스할 수 있다.
- 프라이빗 서브넷(private subnet)<br>
  서브넷에 인터넷 게이트웨이로 직접 연결되는 경로가 없다. 프라이빗 서브넷의 리소스에는 퍼블릭 인터넷에 액세스하기 위해 NAT 디바이스가 필요하다.
<br><br>


**라우팅 테이블**<br>
VPC의 트래픽 컨트롤러 역할을 수행한다. 각 라우팅 테이블에는 라우팅이라는 규칙 세트가 포함되어 있어 서브넷 또는 게이트웨이의 네트워크 트래픽이 전송되는 위치를 결정한다. VPC를 생성할 때 해당 VPC의 기본 라우팅 테이블도 생성된다.
<br>
라우팅 테이블을 사용해 다른 VPC와 온프레미스 네트워크와 같이 VPC가 통신할 수 있는 네트워크를 지정할 수 있다. 

**인터넷 게이트웨이**<br>
VPC 내부의 리소스가 퍼블릭 인터넷과 자유롭게 통신할 수 있도록 해주는 AWS의 관리형 논리 장치(물리적 서버 위의 S/W로 만든 기능)이다. 인터넷 게이트웨이는 AWS 내부에서 자동으로 여러 가용 영역(AZ)에 분산되어 작동하기에, 하나의 가용 영역(AZ)에 장애가 발생하여도 인터넷 연결이 중단되지 않는다. 이러한 특징으로 인해 가용성을 보장하며, 사용자는 별도의 이중화나 유지보수 없이 안정적으로 인터넷 연결을 사용할 수 있다.

**NAT 게이트웨이**<br>
네트워크 주소 변환(NAT) 서비스이다. NAT 게이트를 사용하면 프라이빗 서브넷의 인스턴스가 VPC 외부의 서비스에 연결할 수 있지만 외부 서비스에서 이러한 인스턴스와의 연결을 시작할 수는 없도록 할 수 있다.

**Security Group**<br>
보안 그룹은 인스턴스 수준에서 인바운드와 아웃바운드 트래픽을 제어하는 가상 방화벽으로, 외부와의 통신을 제어한다.
<br>


# 5. AWS Services

## **Computing Service**

**EC2**<br>
Amazon Elastic Compute Cloud의 약자로, AWS에서 제공하는 클라우드 컴퓨팅이다. 독립된 컴퓨터를 임대해주는 서비스로서 AWS의 가장 대표적인 IaaS 상품이다. 우리가 PC방에서 요금을 내고 컴퓨터를 이용하는 것처럼 AWS에서 컴퓨터를 대여해주는 서비스이다.<br>

*특징*
1. 컴퓨팅 요구사항의 변화에 따라 컴퓨팅 파워를 조정할 수 있다.
2. 실제로 사용한 용량만큼만 지불한다.


*장점*
- SLA 약정
    - 신뢰성 및 확장성이 뛰어난 인프라에 온디맨드로 액세스한다.
- AWS Nitro System
    - 애플리케이션을 위한 안전한 컴퓨팅을 제공한다. 보안은 AWS Nitro System에서 Amazon EC2를 기반으로 구축된다.
- 성능 및 비용 최적화
    - AWS Graviton 기반 인스턴스, Amazon EC2 스팟 인스턴스 및 AWS Savings Plans와 같은 유연한 옵션을 통해 성능과 비용을 최적화한다.

**EBS**<br>
스토리지 볼룸을 만들어 Amazon EC2 인스턴스에 연결할 수 있다. 볼륨을 연결한 후에는 해당 볼륨 위에 파일 시스템을 생성하거나 데이터베이스를 실행하거나 블록 스토리지를 사용하는 것과 같은 방식으로 사용할 수 있다.<br>

*장점*
1. 빠른 크기 조정
    - 가장 까다로운 고성능 워크로드에 맞춰 빠르게 확장할 수 있다.
2. 보안
    - 자체 키 관리 인프라를 구축, 유지 및 보호할 필요 없이 블록 스토리지 리소스를 암호화할 수 있다. 퍼블릭 액세스를 제한하고 데이터 백업에 잠금을 구성하여 데이터에 대한 무단 액세스를 방지한다.
3. 간단한 데이터 보호
    - 특정 시점 복사본인 Amazon EBS 스냅샷을 사용하여 클라우드의 블록 스초리지 데이터와 온프레미스 블록 데이터를 보호한다.


**S3**<br>
인터넷용 Object 스토리지로 기본적으로 HTTP와 HTTPS 프로토콜의 인터넷을 통한 액세스 방식을 가진다. 웹에서 언제 어디서나 원하는 양의 데이터를 저장하고 검색할 수 있다. 고도의 확장성과 안정성, 속도 및 내구성을 가지고 있다. S3에 저장할 수 있는 객체 수에 제한이 없고, 저장할 수 있는 객체의 크기는 최대 5TB이다.

**작동 방식**
1. 데이터를 버킷 내에 객체, 계층적 데이터 또는 테이블 형식 데이터로 저장하는 객체 스토리지 서비스이다. 
2. Amazon S3에 데이터를 저장하려면 먼저 버킷을 생성하고 버킷 이름 및 AWS 리전을 지정해야한다. 그런 다음 Amazon S3에서 객체로 해당 버킷에 데이터를 업로드한다. 각 개체에는 키가 있으며, 이는 버킷 내 객체에 대한 고유한 식별자이다.
3. S3은 특정 사용 사례를 지원하도록 구성할 수 있는 기능을 제공한다. 예를 들어, S3 버전 관리를 사용하여 동일한 버킷에 여러 버전의 객체를 보관하고 실수로 삭제되거나 덮어쓰기된 객체를 복원할 수 있다.


- 버킷<br>
  버킷은 S3에 저장된 객체에 대한 컨테이너이다. 버킷의 이름은 리전에 상관없이 전역적으로 고유해야한다.
    - 범용 버킷: 범용 버킷은 대부분의 사용 사례 및 액세스 패턴에 권장되며 S3 버킷 본래의 유형이다. 범용 버킷은 AmazonS3에 저장되는 객체의 컨테이너이다. 버킷과 모든 스토리지 클래스에 원하는 수의 객체를 저장할 수 있으므로 여러 가용 영역에 걸쳐 객체를 중복해서 저장할 수 있다.
    - 디렉터리 버킷: 지연 시간이 짧은 사용 사례 및 데이터 레지던시 사용 사례에 권장된다. 기본적으로 AWS에 최대 100개의 디렉터리 버킷을 생성할 수 있으며, 버킷에 저장할 수 있는 객체 수에는 제한이 없다.
    - 테이블 버킷: 일별 구매 트랜잭션, 스토리밍 센서 데이터 또는 광고 노출과 같은 테이블 형식 데이터를 저장하는데 권장된다. 테이블 형식 데이터는 데이터베이스 테이블과 같이 열과 행의 데이터를 나타낸다. 테이블 버킷은 쿼리 성능을 지속적으로 개선하고 테이블의 스토리지 비용르 절감하도록 설계된 기능이 내장되어 있으며 분석 및 기계 학습 워크로드에 최적화된 S3 스토리지를 제공햔다.
    - 벡터 버킷: S3 벡터 버킷은 벡터를 저장하고 쿼리하기 위해 특별히 구축된 Amazon S3 버킷의 한 유형이다. 벡터 버킷은 전용 API 작업을 사용하여 벡터 데이터를 효율적으로 작성하고 쿼리한다. S3 벡터 버킷을 사용하면 기계 학습 모델의 벡터 임베딩을 저장하고, 유사성 검색을 수행한다.
- 객체(Object)<br>
  객체는 S3에 저장되는 기본 개체이다. 객체는 데이터와 메타 데이터로 구성된다. "데이터"는 AWS S3 console에서는 볼 수 없고, 권한을 갖춘 사용자가 HTTP/HTTPS를 통한 접근 시 볼 수 있다. "메타 데이터"는 객체를 설명하는 key-value의 집합이다. 수정한 날짜, Content-Type과 같은 HTTP 메타 정보 등을 포함한다.
- 객체 버전 관리<br>
  S3는 객체가 중복될 시 덮어쓴다. 이는 실수에 의한 데이터 손실을 유발할 수 있다.
- S3의 보안<br>
  기본적으로 버킷, 객체 및 하위 리소스는 모두 비공개이다. 즉, 리소스를 만든 계정 소유자만 액세스할 수 있다. S3는 리소스 기반 정책과 사용자 정책으로 접근에 대해 정의한다.

**EFS**<br>
파일 스토리지 서비스로 여러 EC2 인스턴스가 파일을 공유할 수 있다.<br>
**특징**
1. 스케일링
    - 수천 개의 EC2에서 동시에 액세스 가능하며 탄력적으로 파일을 추가하고 삭제함에 따라 자동으로 Auto Scaling이 가능하다. 즉, 미리 크기를 프로비저닝할 필요가 없다.
    - 페타바이트단위 데이터로까지 확장 가능하다.
2. 가용성
    - VPC 내에서 생성되며, 파일 시스템 인터페이스를 통해 EC2에 액세스한다.
    - 여러 가용영역에서 액세스 가능하다.
    - 여러 가용영역에 중복 저장되기 때문에 하나의 가용영역이 파괴되더라도 다른 AZ에서 서비스 제공 간으하다.
3. 스토리지 클래스
    - 스토리지 클래스가 존재한다.
4. 수명 주기 관리
5. 파일 마운트
    - EFS DNS 이름으로 mount 해서 여러 인스턴스에서 동시에 사용할 수 있다.
    - Block Storage와 마찬가지로 File storage도 mount 하면 내 로컬의 파일 시스템처럼 사용할 수 있다.
    - Amazon EFS를 사용하여 기존 파일 시스템에서 AWS DataSync로 파일을 전송할 수 있다.
- 스토리지 클래스: 자주 액세스하는 파일을 저장하는데 사용되는 클래스

**장점**
1. 빠르고 손쉬운 생성
    - AWS 컴퓨팅 시스템을 위한 공유 파일 시스템을 간편하고 빠르게 생성하고 구성한다.
2. 탄력적이고 확장 가능
    - 온디맨드 워크로드를 페타바이트 규모의 스토리지 및 초당 기가바이트의 처리량으로 즉시 확장할 수 있다.

**Route 53**<br>
DNS로, 사용자의 도메인을 관리하고 트래픽을 적절한 서버로 라우팅하는 서비스이다. 고가용성과 확장성을 제공하며 전 세계적인 분산 DNS 인프라를 제공한다.

**특징**
- 가용성과 확장성이 뛰어난 클라우드 기반 DNS 웹 서비스이다.
- 동적으로 사용자에게 노출된 DNS 레코드 타입과 값을 조정한다.
- 각종 다양한 로드밸런싱 기능을 지원한다.
- 사용자의 요청을 EC2, ELB, S3 버킷 등 인프라로 직접 연결 간으하다.
- 외부의 인프라로 라우팅하는데 사용 가능하다.
- Route 53 트래픽 흐름을 사용하면 지연시간기반 라우팅이 가능하다.
- Route 53에서는 도메인 이름 등록도 지원 가능하다.

**ELB**
로드 밸런서로, 여러 EC2 인스턴스에 트래픽을 분산시킨다.
- 애플리케이션의 가용성을 높이고 부하에 따라 자동으로 트래픽을 분배한다.

**구성 요소**
- 리스너(Listener): 외부 요청을 받아들이는 역할을 한다. 서비스 포트를 통해 요청을 수신하며 요청이 유효하다면 대상 그룹으로 전달한다. 리스너는 포트 변환을 통해 들어온 요청을 다른 포트로 전달할 수도 있다.
- 대상 그룹(Target Group): 트래픽이 분산될 리소스들의 집합이다. EC2 인스턴스, 컨테이너, IP주소 등이 포함될 수 있으며, 각 대상은 서비스 포트를 열어 둬야 한다. ELB는 대상 그룹에 등록된 리소스의 상태를 Health Check로 지속적으로 확인한다. 정상 상태인 대상에게만 요청이 전달된다.
- 상태 체크(Health Check): 대상 그룹의 EC2의 상태를 체크하는 것이다.
- 유휴 제한 시간(Connection Time Out): 사용자가 ELB를 거쳐 EC2에 접근하여 서비스를 접속하면 Connection이 생성된다. 그리고 이 Connection을 통해 사용자와 EC2가 통신을 한다. 그리고 더 이상의 통신이 없을때 유휴 제한 시간이 작동하고, 그 시간이 지나면 Connection이 사라진다.

**요청 처리 과정**
1. 사용자가 로드밸런서에 접근하기 위해 Amazon의 DNS 서버에 로드밸런서의 도메인 해석을 요청
2. AMazon의 DNS 서버가 로드밸런서 노드 IP 리스트를 사용자에게 전달
3. 사용자는 IP 중 하나를 선택하여 로드밸런서에 접근( +Port를 입력)
4. 사용자는 로드밸런서의 Port가 일치하는 리스너에 접근하게 되며 리스너는 이 요청을 받아들여 적절한 대상 그룹으로 전달
6. 리스너로부터 전달받은 요청을 EC2가 처리한 후 다시 사용자에게 반환

*종류*
1. ALB(Application Load Balancer)
    - OSI 7계층에 해당하는 Application Layter의 특성을 이용하는 로드밸런서이다. 그중에서도 HTTP, HTTPS의 특성을 주로 다룬다. 다시 말해 단순 부하분산 뿐만 아니라 HTTP 헤더 정보를 이용해 부하분산을 처리한다. 경로를 기반으로 라우팅하고, 트래픽 처리에 최적화되어있다. 고정 IP 주소를 사용하지 못한다.
2. NLB(Network Load Balancer)
    - OSI 4계층에 해당하는 Transport Layer의 특성을 이용하는 로드밸런서이다. 즉 TCP와 UDP를 사용하는 요청을 받아들여 부하분산을 실시한다. 예를 들어 HTTP도 TCP 기반의 프로토콜이기 때문에 ALB를 사용하지 않고 NLB를 사용한다면 이를 받아들여 부하분산을 실시한다. 다만 상위 계층인 HTTP가 아닌 하위 계층의 TCP 계층에서의 처리이므로 HTTP 헤더를 해석하지 못한다.
3. GLB(Gateway Load Balancer)
    - OSI 3계층에서 동작하며 보안 장비(Virtual Appliance)와 함께 사용한다. 트래픽을 투명하게 전달하고 모니터링이 가능하다.
4. CLB(classic Load Balancer)
    - Elastic Load Balancing 이전 세대의 로드 밸런서이다. OSI 4계층과 OSI 7계층에서 동작한다. HTTP/HTTPS/TCP/SSL의 트래픽을 처리한다. 현재는 사용하지 않는다.

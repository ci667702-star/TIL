# Networking

## 1. IPv4, IPv6
ipv4와 ipv6은 모두 TCP/IP(Transmission Control Protocol/internet pRotocol)프로토콜 제품군에 속한다.

**IP주소란?**<br>
    인터넷 세상에서 통신하기 위해 각각의 컴퓨터 및 통신장비에 부여하는 고유한 주소를 의미한다.<br>
    IP주소는 네트워크ID와 호스트ID로 나누어져있다.
    네트워크ID는 네트워크를 구분해주는 ID를 말하고 기기에 부여하는 고유 번호를 호스트ID라고 한다

**IPv4 주소**<br>
인터넷 상에서 컴퓨터 간의 통신을 가능하게 하는 프로토콜이다. 
IPv4 주소는 32비트 길이의 식별자로 0.0.0.0~255.255.255.255까지의 숫자의 조합으로 이루어지며 총 네구간으로 나누어져있고 비트 단위를 점(.)으로 구분한다. IPv4를 통해 최대 약 43억개의 서로 다른 주소를 부여할 수 있다. 하지만 전세계 공용으로 사용되며 인터넷 사용자수가 급증하면서 IPv4주소가 고갈될 문제에 처하자 이를 해결하기 위해 IPv6 주소가 등장했다.

**IPv6 주소**<br>
IPv6 주소 체계를 128비트 크기로 확장한 차세대 인터넷 프로토콜 주소이다. 일반적으로 16비트 단위로 나누어지며 각 16비트 블록은 다시 4자리 16진수로 변환되고 콜론(:)으로 구분되어진다.

구분 | IPv4 | Ipv6
:--:|:--:|:--:
주소 길이 | 32비트 | 128비트
표시 방법 | 8비트씩 4부분으로 <br>10진수로 표시 | 16비트씩 8부분으로 <br> 16진수로 표시
주소 개수 | 약 43억개 | 약 43억 x 43억 x 43억 x 43억
주소 할상 | A,B,C 등 클래스 단위의 비순차적 할당 | 네트워크 규모 및 단말기 수에 따른 순차적 할당
품질 제어 | 지원 수단 없음 | 등급별, 서비스별로 패킷을 구분할 수 있어 품질보장이 용이함
보안 기능 | IPsec 프로토콜 별도 설치 | 확장기능에서 기본으로 제공
PnP| 지원 수단 없음 | 지원 수단 있음
모바일 IP | 상당히 곤란 | 용이
웹캐스팅 | 곤란 | 용이


## 2. ARP/RARP
**ARO**<br>
Address Resolution Protocol의 약자로 네트워크 계층의 주소 결정 프로토콜이다. 호스트의 IP주소를 호스트와 연결된 네트워크 접속 장치의 물리적 주소로 바꾼다.

**ARP의 동작 방식**
- ARP는 브로드캐스트로 동작한다.
- CPU에서 직접 수행한다.

네트워크망 내의 모든 host에게 브로드캐스트로 패킷을 보내면 목적지가 해당 패킷에 자신의 MAC 주소를 실어서 유니캐스트로 응답하고 ARP는 이 정보를 테이블에 저장한다.

**RARP** <br>
Reverse Address Resolution Protocol의 약자이다.
ARP의 반대로 동작하는 것인데 ARP 프로토콜 구조는 같지만 필드의 내용이 다르고 원래 목적과 반대로 사용된다.(물리적 주소를 IP주소로 변환한다.)

- IP주소가 정해져 있지 않은 단말이 IP 할당을 요청할 떄 사용한다.
- 자신의 MAC 주소는 알지만 IP가 할당되지 않아 IP를 할당해주는 서버에게 어떤 IP주소를 사용해야 하는지 물어볼 떄 사용된다.

## 3. 3wayhandshake

**3-way handshake** <br>
전송 제어 프로토콜(TCP)에서 통신하는 장치 간 연결이 잘 되었는지 확인하는 과정/방식이다.
TCP/IP 4계층을 통과, 데이터를 보낼 준비가 되면 수신 측이 받을 준비가 되었는지 확인하는 과정이다.
3-way handshake를 수행하기 위해서는 SYN과 ACK 플래그들이 사용된다.

**3-way handshake 과정** <br>
1. 클라이언트 -> 서버에 접속 요청 메시지 SYN 패킷 전송
    - `SYN (Synchronize Sequence Number): 임의의 랜덤 숫자`
2. 서버 -> SYN 받음 -> 클라이언트에게 연결 요청 메세지 ACK + SYN 패킷 전송
    - `ACK(Acknowledgement number): SYN+1의 값 `
3. 클라이언트 -> ACK + SYN 받음 -> 서버에 ACK 전송
=> 3번의 통신 완료 후, 연결 성립


## 4. 4wayhandshake

**4-way handshake**<br>
4-way handshake은 연결을 해제(Connection Termination)하는 과정이다. 여기서는 FIN 프래그를 사용한다.
- `FIN(finish): 세션을 종료시키는데 사용되며 더 이상 보낸 데이터가 없음을 나타낸다.`

**4-way handshake 과정** 
1. 클라이언트 -> 서버에게 연결 종료를 알리는 FIN플래그가 설정된 패킷 전송.
2. 서버 -> FIN 신호 확인 -> 확인했다고 알려주기 위해 ACK 신호 전송.
3. 서버 통신 끝 -> 연결 종료 요청 합의의 의미로 클라이언트에게 FIN 플래그가 설정된 패킷 전송.
4. 클라이언트 -> FIN 받음 -> 확인의 의미로 ACK 플래그가 설정된 패킷 보냄
=> 4번의 통신 이후 연결 해제

**3-way handshake와 4-way handshake의 차이점**<br>
3-way handshake는 TCP 연결을 생성하는 데 사용되고, 4-way handshake는 TCP 연결을 종료하는 데에 사용된다.


## 5. 브로드캐스트, 유니캐스트, 애니캐스트

**브로드캐스트** <br>
1:n의 관계를 가진다.
로컬 랜 상에 붙어있는 (브로드캐스트 도메인 안에 있는) 모든 네트워크 장비들에게 보내는 통신이다.
브로드 캐스트의 주소는 FFFF.FFFF.FFFF(맥 어드레스일 경우)이다.
이 주소로 패킷을 CPU가 받으면 무조건 읽어들인다.
브로드캐스트는 네트워크 상의 전체 노드로 전송되기 때문에 전체 트래픽이 증가한다.
이 패킷을 받은 CPU는 이 패킷을 처리하게 되고 PC의 성능도 떨어진다.
즉, 과도한 브로드캐스트는 전체 네트워크 성능 뿐만 아니라 PC의 성능도 떨어지게 한다.

**유니캐스트**<br>
1:1의 관계를 가진다.
정보를 전송하기 위한 프레임에 자신의 MAC 주소와 목적지의 MAC 주소를 첨부하여 전송하는 방식이다. 어떤 시스템이 유니캐스트 방식으로 데이터를 전송하게 되면 같은 네트워크에 있는 모든 시스템들은 그 MAC 주소를 받아서 자신의 MAC 주소와 비교 후에 자신의 MAC 주소와 같지 않다면 프레임을 버리고 같다면 프레임을 받아서 처리하게 된다. 유니캐스트 방식은 가장 많이 사용하는 방식으로 한 개의 목적지 MAC주소를 사용하고 CPU 성능에 문제를 주지 않는 방식이다.

**멀티캐스트**<br>
네트워크에 연결되어 있는 시스템 중 일부에게만 정보를 전송하는 것으로 특정 그룹에 속해 있는 시스템에게만 한 번에 정보를 전송할 수 있는 방법을 말한다. 멀티캐스트는 라우터가 멀티캐스트를 지원해야만 사용 가능하다는 단점이 있다. 티캐스트 전송이 지원되면 송신자는 여러 수신자에게 한 번에 메시지가 전송되도록 하여 데이터의 중복 전송으로 인한 네트워크 자원 낭비를 최소화할 수 있게 된다.

- 멀티캐스트 그룹 단위로 묶어 그 그룹의 Host 들은 동시에 데이터를 받을 수 있다.
- UDP를 사용하여 전송함으로 신뢰성을 보장받지는 못한다.
- Client에서 멀티캐스트를 사용하는 Application 을 시작하면 멀티캐스트 IP주소오 멀티캐스트 MAC주소를 라우터에 등록함으로 멀티캐스트 그룹에 등록된다.
- 하나의 Client 에서 여러 멀티캐스트 주소를 수용할 수 있다. 즉 여러가지 멀티캐스트 데이터를 동시에 받을 수 있다.

**애니캐스트**<br>
애니캐스트는 여러 수신자 중 가장 가까운 수신자로 트래픽을 전달하는 방식이다. IPv4에서는 애니캐스트 주소를 유니캐스트 주소와 동일한 형식으로 사용하며, 주로 DNS 서버나 콘텐츠 배포 네트워크에서 사용된다.
IPv6에서는 더 발전된 방식으로 지원된다.
## 6. TCP/UDP
**TCP**<br>
TCP를 해석하면 전송을 제어하는 프로토콜이라는 뜻인으로, 인터넷상에서 데이터를 메세지의 형태로 보내기 위해 IP와 함께 사용하는 프로토콜입니다.

**특징**
1. 연결 지향 방식으로 패킷 교환 방식을 사용한다.
    - `연결 지향 방식: 1:1 연결 상태를 유지하여 통신하는 것을 말한다.`
2. 3-way handshaking 과정을 통해 연결을 설정하고 4-way handshaking을 통해 해제한다.
3. 흐름 제어 및 혼잡 제어
4. 높은 신뢰성을 보장한다.
5. UDP보다 속도가 느리다.
6. 전이중, 점대점 방식을 사용한다.

**TCP 통신 과정**<br>
1. 연결 확립
    - 3-way handshake
2. 데이터 전송
    - 3-way handshake로 송신 호스트와 수신 호스트 사이에 연결 확립이 되면, 데이터를 전송할 수 있는 연결 통로가 만들어지게 된다.
3. 연결 해제
    - 데이터 전송이 완료된 후에는 연결을 끊기 위한 요청을 교환해야한다.

- `패킷이란?`
인터넷 내에서 데이터를 보내기 위한 경로배정을 효율적으로 하기 위해서 데이터를 여러 개의 조각들로 나누어 전송하는데, 이때 이 조각을 패킷이라고 합니다.

**UDP**<br>
해석하면 사용자 데이터그램 프로토콜이라는 뜻으로, 데이터를 데이터그램 단위로 처리하는 프로토콜이다.

**특징**
1. 비연결형 서비스로 데이터그램 방식을 제공한다.
2. 정보를 주고 받을 떄 정보를 보내거나 받는다는 신호절차를 거치지 않는다.
3. 신뢰성이 낮다.
4. TCP보다 속도가 빠르다.

UDP는 비연결형 서비스이기 때문에 연결을 설정하고 해제하는 과정이 존재하지 않는다.

## 7. OSI 7layer
**응용 계층 > 표현 계층 > 세션 계층 > 전송 계층 > 네트워크 계층 > 데이터링크 계층 > 물리 계층**

7개의 Layer로 구성되어 있고, 7~3 계층까지는 변화가 발생하지 않는다.
하지만 2계층 부터는 데이터를 전송하는 방식에 따라 변화가 발생할 수 있다.

구분 | 계층 | 기능| 주요 프로토콜 및 장비
:--: | :--: | :--: | :--:
7 | 응용 계층 | 사용자에게 네트워크 서비스 제공 | HTTP, FTP, DNS, SMTP, POP3
6 | 표현 계층 | 데이터 변환(인코딩, 압축, 암호화) | JPEG, PNG, SSL, TLS
5 | 세션 계층 | 연결 설정 및 유지, 세션 관리 | PRC, NetBIOS
4| 전송 계층 | 신뢰성 있는 데이터 전송 port 번호 | TCP-세그멘테이션 작동<br>UDP-데이터 자르지 않음
3| 네트워크 계층 | IP 주소 기반 데이터 전달 | IP, ICMP, ARP, 라우타
2 | 데이터 링크 계층 | MAC주소 기반 데이터 전달 | Ethernet, MAC 주소, 스위치
1 | 물리 계층 | 전기적 신호, 물리적 매체 | 케이블, 리피터, 허브
- 7~5 계층 -> 사용자와 가까운 부분이다.
- 4~3 계층 -> 데이터 전송을 관리한다.
- 2~1 -> 하드웨어 기반 데이터를 전송한다.

**특징**<br>
각 레이어는 캡슐화 되어있기 떄문에 서로 독립적으로 작동한다.
하나의 레이어에서 문제가 발생하더라도 다른 레이어에 영향을 주지 않는다.
    - `캡슐화: 객체 지향 프로그래밍에서 데이터와 그 데이터를 처리하는 기능을 하나로 묶고, 외부에서 상세 내부 구현을 숨겨 직접 접근을 막는 기술이다.`
    
## 8. 브리지

**브리지**<br>
네트워크 장비 중 하나로, 두 개 이상의 네트워크 세그먼트를 연결하여 데이터를 전달하고 네트워크 트래픽을 관리하는 역할을 한다. 브리지는 네트워크 프로토콜을 사용하는 네트워크 세그먼트를 연결하며, 데이터가 필요하지 않은 세그먼트로는 전송되지 않도록 필터링 기능을 제공한다.
OSI2계층에서 동작한다.

- `세그먼트(segment): 하나의 네트워크를 브리지 등을 사용해서 분리했을 때 그 각각을 지칭하는 단위이다.`

**특징**<br>
1. 작동 계층
    - OSI 모델의 2계층에서 동작하며, 네트워크 데이터 패킷의 MAC 주소를 기반으로 처리.

2. 충돌 도메인 분리
    - 네트워크를 여러 세그먼트로 나누어 각 세그먼트의 충돌 도메인을 분리함으로써 네트워크 트래픽이 감소하고 보안성도 증가하게 된다.

3. 데이터 필터링
    - 데이터를 목적지 MAC 주소를 기준으로 필터링하여 해당 데이터를 필요한 세그먼트로만 전달.

4. 효율성 증가
    - 불필요한 데이터 전송을 줄여 네트워크 혼잡을 완화.

5. 프로토콜 독립적
    - 동일한 데이터 링크 계층 프로토콜을 사용하는 네트워크를 연결.

**역할**<br>
1. 네트워크 세그먼트 연결
    - 동일한 네트워크 프로토콜을 사용하는 두 개 이상의 네트워크를 연결하여 하나의 네트워크처럼 동작.
2. 트래픽 필터링
    - 데이터를 목적지 MAC 주소와 비교하여 필요하지 않은 네트워크 세그먼트로 데이터가 전달되지 않도록 차단.
3. 네트워크 성능 향상
    - 네트워크 세그먼트를 나누어 충돌 도메인을 분리함으로써 혼잡을 줄이고 성능을 최적화.
4. 브로드캐스트 관리
    - 네트워크 상의 브로드캐스트 트래픽을 제한하여 과도한 트래픽을 방지.

## 9. 스위치

**스위치**
컴퓨터와 같은 둘 이상의 IT 디바이스가 서로 통신할 수 있도록 통신 네트워크를 구성하는 기술이다. 스위치는 네트워크의 디바이스가 프린터, 서버, 파일 스토리지, 인터넷 접속 및 처리 등 IT 리소스를 공유할 수 있도록 한다. 간단한 스위치는 단일 LAN에 디바이스를 연결하는 반면, 고급스위치는 여러 LAN에서 디바이스를 연결하고 데이터 보안 기능을 통합할 수 있다.

**역할**
- 네트워크의 중재자
  네트워크 중간에서 패킷을 받아 목적지에게만 보내준다.
- 동시 통신, 패킷 충돌 X
여러 장비가 서로 간섭 및 패킷 충돌 없이 동시에 통신하도록 도와준다.
- 네트워크 효율성 UP
통신을 위해 기다리거나 충돌로 인해 대기하는 문제를 방지하여 네트워크 전체의 통신 효율성이 향상된다.
- 어떤 장비가 어느 위치에 있는지 파악하고 통신이 시작되면 자신이 알고 있는 위치로 패킷을 정확히 전송한다.
    이를 위해 MAC 주소 테이블을 가지고 있다.
<br><br>

**스위치 작동 방법**<br>
1. Flooding
    - 플러딩은 스위치에 연결된 모든 포트로 패킷을 전달하는 방식을 말한다.
    스위치는 처음 부팅하면 MAC 주소 테이블에 네트워크 관련 정보가 아무것도 없어 허브처럼 동작한다.
        > 패킷이 스위치에 들어오면 해당 패킷의 MAC 주소를 보고 이를 학습해 MAC 테이블에 저장한다.
    - TCP/IP에서 ARP 브로드캐스트를 미리 주고받은 뒤 데이터가 전달되어 실제 데이터를 보내고 받을 때는 플러딩이 발생하지 않으며, 필요한 곳에만 패킷을 포워딩하게 된다.

2. Address Learning
    - 스위치가 목적지 포트로 정확히 포워딩하려면 MAC 주소 테이블을 만들고 올바르게 유지해야 하는데 이 과정을 어드레스 러닝이라고 한다. 스위치는 일단 패킷을 받으면 받은 패킷과 포트 번호를 MAC 테이블에 작성한다.

    러닝 과정<br>

        1. 스위치에 [ 출발지 MAC (AA-AA-AA), 스위치 Port : 1 ] 정보를 가진 패킷이 들어왔다.<br>
        2. 스위치는 MAC 주소 테이블에 출발지 MAC주소와 포트번호를 기록한다.<br>
        3. 기록된 정보를 가지고 스위치 1번 포트에 AA-AA-AA주소를 가진 장비가 붙어 있다는 것을 습득한다.

3. Forwarding/Filtering

    - 포워딩과 필터링은 스위치의 필수 기능이다.

    - 패킷이 스위치에 들어온 경우 목적지 MAC를 확인한 후 자신이 가진 정보와 매치되는 포트로만 넘겨주는 것을 포워딩이라고 하며, 다른 포트로는 패킷을 전송하지 않는 것을 필터링이라고 한다.
    - 일반적인 유니캐스트 통신에서만 포워딩과 필터링 작업이 모두 수행되며, 멀티/브로드캐스트는 목적지 주소가 특정 단말이 아니기 때문에 포워딩/필터링이 불가능하여 플러딩된다.

4. Aging
    - 스위치에서 MAC adress는 일정시간이 지나면 삭제된다.
    - Aging Timer가 끝나기 전에 해당 포트로 frame이 들어오면 처음부터 다시 카운트된다.
    - MAC adress table의 효율적 관리를 위해 사용된다.

    -`프레임: 데이터 링크 계층에서 사용되는 패킷이다.`

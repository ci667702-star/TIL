# AWS

# **퍼블릭 클라우드(Public Cloud)**
여러 기업과 개인이 같은 클라우드 인프라를 공유해서 쓰는 모델이다.. 논리적으로는 철저하게 분리된다. 즉, 아파트처럼 공유 인프라를 사용하여 비용 효율적이다.
    
**장점**
1. 비용 절감
    - 사용자가 필요한 만큼의 자원을 이용할 수 있도록 유연하게 제공함으로써 비용을 절감할 수 있다.
2. 높은 가용성과 확장성을 보장
    - 여러 지역에 데이터 센터를 구축하고, 여러 대의 서버를 구성하여 서비스의 가용성을 높인다.
3. 전문적인 관리 서비스를 제공
    - 자원의 업데이트와 유지 보수를 책임지기 때문에 사용자는 이에 대한 부담을 덜 수 있다.
4. 사용이 쉽고 편리
    - 인터넷을 통해 제공되므로 사용자가 전 세계 어디에서든 인터넷에 연결되어 있다면 언제든지 이용할 수 있다.
5. 혁신적인 기술에 대한 접근성을 높일 수 있음
    - 인공지능, 머신러닝, 빅데이터 분석, IoT 등 최신 기술을 쉽게 활용할 수 있다. 개별 기업이 직접 구축하기 어려운 고비용의 첨단 기술을 합리적인 비용으로 이용할 수 있다.

**단점**
1. 보안
    - 데이터가 인터넷을 통해 전송되고 저장되기 때문에 해커의 공격에 노출될 수 있다.
2. 시스템 및 네트워크의 제한
    - 제공 업체가 제어하기 때문에 사용자는 네트워크 구성과 관련된 제한을 받게 된다. 그리고 인터넷 연결이 필요하다.

# **프라이빗 클라우드(Private Cloud)**
하나의 조직에 전용 클라우드 환경을 제공하는 모델이다. 사내 전용 클라우드로써, 각 조직이 더 효율적으로 관리할 수 있으나, 퍼블릭 클라우드에 비해 비용이 많이 든다. 즉, 단독주택처럼 전용 인프라로 보안성과 통제력을 확보한다. 각 조직에 on-premise 방식으로 적용될 수도 있고, 퍼블릭 클라우드 제공업체에서 off-premise로 호스팅 될 수도 있다. 

**특징**
- 전용 인프라를 사용하여 물리적, 가상적으로 분리된 환경에서 운영된다.

**장점**
1. 안정성과 보안성이 높다
    - 전용 인프라를 사용하므로 높은 안정성과 보안성을 보장한다.
2. 성능이 우수하고 유연성이 높다.
    - 기업이나 조직이 직접 IT 인프라를 관리하기 때문에 성능이 우수하고 사용자 제어권이 높다. 업무 처리 속도가 빨라지고 생산성이 향상될 수 있으며 기업이나 조직은 IT 자원의 사용과 관리를 더욱 효율적으로 할 수 있다. 또한 기업이나 조직의 요구에 맞게 설계되므로 유연성이 높다.

**단점**
1. 초기 투자 및 운영 비용이 많이 듦
    - 전용 인프라를 구축해야 하므로 초기 비용이 많이 든다. 또한 운영을 위한 전문 기술과 인력이 필요하며 사용자 수나 데이터양이 적을 경우에는 공용 클라우드보다 운영 비용이 높을 수 있다.
2. 인프라 운영의 책임과 위험성
    - 전용 인프라를 사용하기 때문에 확장성에 제한이 있을 수 있다. 또한 기업이나 조직은 고장 대비 계획을 수립하고 유지보수를 진행해야한다.

# **하이브리드 클라우드(Hybrid Cloud)**
VPN등의 보안연결을 통해서 하나 이상의 퍼블릭 클라우드 + 프라이빗 클라우드 환경을 결합하는 방식이다. 두 개의 클라우드 간의 트래픽은 유동적으로 조정이 가능하여 유연한 대처가 가능하다.

**장점**
1. 보안성
    - 기업은 중요한 데이터나 애플리케이션은 프라이빗 클라우드에, 가벼운 작업은 퍼블릭 클라우드에 배치할 수 있다. 이를 통하여 중요한 데이터와 애플리케이션을 프라이빗 클라우드에 유지할 수 있으므로 퍼블릭 클라우드에서 발생할 수 있는 보안 위협을 줄일 수 있다. 
2. 유연성과 확장성
    - 기업은 비즈니스 요구 사항에 따라 리소스를 즉시 확장하거나 축소할 수 있다.
3. 비용 절감
    - 기업은 클라우드에서 필요한만큼 리소스를 구매하여 비용을 절감할 수 있다.

**단점**
1. 운영 및 관리의 어려움
    - 퍼블릭 클라우드와 프라이빗 클라우드를 모두 운영하므로 관리 복잡성이 증가할 수 있다.
2. 성능과 호환성 문제
    - 서로 다른 클라우드 간에 데이터 이전에 필요할 때 데이터와 애플리케이션 간의 통신이 느려지는 등의 성능 문제를 유발할 수 있다.



# IAM
IAM(Identity and Access Management)은 AWS 리소스에 대한 액세스를 안전하게 제어할 수 있는 웹 서비스이다. IAM을 사용하면 사용자가 액세스할 수 있는 AWS 리소스를 제어하는 권한을 관리할 수 있다. IAM을 사용하여 리소스를 사용하도록 인증(로그인) 및 권한 부여(권한 있음)된 대상을 제어한다.

**특징**
1. 강력한 사용자 인증
    - 콘솔 및 API 접근 시 사용자는 다중 인증 및 인증키 인증을 반드시 수행해야한다.
2. 손쉬운 권한 관리
    - Account 관리자(Root 사용자)는 사용자 및 사용자 그룹, 정책, 역할을 손쉽게 생성하여 관리할 수 있다. 업무 목적에 따라 사용자(IAM 사용자)를 그룹 및 역할에 매핑하여 사용자의 접근 권한을 업무에 필요한 부분으로 한정한다.
3. 임시 자격증명을 통한 접근 통제
    - Account 관리자는 임시 자격증명을 통한 역할을 생성하여 필요에 따라 최소한의 권한만 부여할 수 있다. 임시 자격증명을 통해 제한된 시간동안 필요한 권한만을 제공함으로써, 다수의 사용자에 대해 높은 수준의 중앙집중식 관리가 가능하다.

**구성**
- 관리
    - 시스템에서 사용자 ID를 생성, 유지 관리 및 안전하게 폐기하는 프로세스이다.
- 인증
    - 사용자가 자신이 주장하는 사람과 동일하다는 것을 확인하는 과정이다
- 권한 부여
    - 확인된 사용자에게 리소스에 대한 적절한 수준의 액세스 권한을 부여하는 프로세스이다.

# **Auto Scaling**
시스템 요구에 따라 컴퓨팅 리소스를 자동으로 할당하는 클라우드 컴퓨팅 기능이다. 애플리케이션이 일관된 가용성을 유지하고 성능 목표를 달성하는 데 필요한 리소스를 확보하고, 클라우드 리소스의 효율적인 사용을 촉진하고 클라우드 비용을 최소화하는 데 사용된다. 

**장점**
1. 인프라의 수동 구성 최소화
    - Auto Scaling을 사용하면 사람이 개입할 필요 없이 클라우드 환경이 리소스 수요에 실시간으로 대응할 수 있다. 
2. 확장성 중대
    - Auto Scaling을 통해 조직은 리소스 모니터링 및 프로비저닝에 추가 인력을 투입할 필요 없이 클라우드 컴퓨팅 환경과 기능을 더욱 원활하게 확장할 수 있다.
3. 일관된 성능 제공
    - Auto Scaling은 활동이나 수요에 관계없이 클라우드 환경에 필요한 컴퓨팅, 네트워크 및 스토리지 리소스를 확보함으로써 클라우드 서비스의 일관되고 안정적인 성능을 유지하는 데 도움이 된다.
4. 사용자 경험 개선
    - 보다 일관된 웹 앱 및 네트워크 성능은 사용자에게 보다 일관된 수준의 서비스를 제공한다는 것을 의미한다.
5. 클라우드 컴퓨팅 비용 절감
    - 리소스의 수동 프로비저닝에 의존하는 경우, 조직은 수요가 급증하는 시기에 리소스를 사용할 수 있도록 하기 위해 예방 차원에서 오버프로비저닝을 하는 경우가 많다. 수요에 따라 컴퓨팅, 네트워크, 스토리지 자원을 실시간으로 자동 확장할 수 있는 플랫폼을 사용하면 조직은 리소스 과잉 할당을 피할 수 있다.

# **ELB**
ELB(Elastic Load Balancer)는 AWS의 로드 밸런서 서비스이다. 로드밸런서의 주요 기능은 서버 부하 분산, 즉 로드밸런싱으로 특정 서버에 부하가 몰리지 않도록 적절히 분산하는 것이다. EC2 뿐만 아니라 ECS의 컨테이너, Lamba 등을 상대로 부하분산을 실시한다.

**장점**
1. 보안
    - SSL/TLS 종료, 통합 인증서 관리, 클라이언트 인증서 인증을 통해 애플리케이션을 보호한다.
2. 자동 조정
    - 고가용성 및 자동 크기 조정으로 애플리케이션을 지원한다.
3. 실시간 모니터링
    - 실시간으로 애플리케이션 상태 및 성능을 모니터링하고 병목 현상을 파악하며 SLA 규정 준수 상태를 유지관리한다.

**종류**
1. Application Load Balancer(ALB)
    - OSI 7계층에 해당하는 Application Layer의 특성을 이용하는 로드밸런서이다. 그중에서도 HTTP, HTTPS의 특성을 주로 다루는 로드밸런서이다. 다시 말해 단순 부하분산 뿐만 아니라 HTTP 헤더 정보를 이용해 부하분산을 실시하는 것이 ALB의 가장 중요한 특징이다.
2. Network Load Balancer(NLB)
    - OSI 4계층에 대당하는 Transport Layer의 특성을 이용하는 로드밸런서이다. 즉 TCP와 UDP를 사용하는 요청을 받아들여 부하분산을 실시한다. 예를 들어 HTTP도 TCP 기반의 프로토콜이기 때문에 ALB를 사용하지 않고 NLB를 사용한다면 이를 받아들여 부하 분산을 실시한다. 다만 상위 계층인 HTTP가 아닌 하위 계층의 TCP 계층에서의 처리이므로 HTTP 헤더를 해석하지 못한다.

**구성 요소**
- 리스너(Listener)
    - 리스너는 사용자의 요청을 받아들이고 적절한 대상 그룹으로 라우팅하는 역할을 한다. 
- 대상 그룹(Target Group)
    - 대상 그룹은 리스너가 전달한 요청을 처리하기 위한 부하분산 대상들의 모임이다.
- 보안 그룹(Security Group)
    - ELB또한 EC2처럼 보안 그룹을 지닐 수 있다. 가질 수 있는 이유는 로드밸런서 노드 또한 네트워크 인터페이스의 형태를 띄기 때문이다.
- 상태 체크(Health Check)
    - 대상 그룹의 EC2의 상태를 체크하는 것을 말한다.
- 유휴 제한 시간(Connection Time Out)
    - 사용자가 ELB를 거쳐 EC2에 접근하여 서비스를 접속하면 Connection이 생성된다. 그리고 이 Connection을 통해 사용자와 EC2가 통신을 한다. 그리고 더 이상의 통신이 없을 때 Connection Time Out이 작동하고, 그 시간이 지나면 커넥션이 사라진다.

**요청 처리 과정**
1. 사용자가 로드밸런서에 접근하기 위해 Amazon의 DNS 서버에 로드밸런서의 도메인 해석을 요청
2. AMazon의 DNS 서버가 로드밸런서 노드 IP 리스트를 사용자에게 전달
3. 사용자는 IP 중 하나를 선택하여 로드밸런서에 접근( +Port를 입력)
4. 사용자는 로드밸런서의 Port가 일치하는 리스너에 접근하게 되며 리스너는 이 요청을 받아들여 적절한 대상 그룹으로 전달
5. 리스너로부터 전달받은 요청을 EC2가 처리한 후 다시 사용자에게 반환

# RDS
RDS(Relational Database Service)는 총 소유 비용에 최적화되고 쉽게 관리할 수 있는 관계형 데이터베이스 서비스이다.

- 관계형 데이터베이스: 관계형 데이터베이스는 데이터를 행과 열로 구성하는 데이터베이스 유형으로, 데이터 포인트가 서로 관련된 테이블을 집합적으로 형성한다.

**특징**
- 사용 편의성
    - 몇 분 이내에 데이터베이스 인스턴스를 시작하고 애플리케이션을 연결할 수 있다. DB 파라미터 그룹을 사용하면 데이터베이스를 세부적으로 제어하고 튜닝할 수도 있다.
- 자동 소프트웨어 패치
    - RDS는 최신 패치를 통해 배포를 지원하는 관계형 데이터베이스 소프트웨어가 최신 상태로 유지되도록 한다. 데이터베이스 인스턴스의 패치 여부와 시기를 선택적으로 제어할 수 있다.
- 모범 사례 권장 시스템
    - RDS에서는 데이터베이스 인스턴스의 구성과 사용 지표를 분석하여 모범 사례 지침을 제공한다.
- 읽기 전용 복제본 시스템
    - DB 인스턴스의 복제본을 하나 이상 생성하여 대량의 애플리케이션 읽기 프래픽을 처리할 수 있는 기능을 제공한다.
- 다중 AZ 구성 배포 
    - 다중 AZ 구성은 현재 서비스되는 RDB에 문제가 되어도 다른 AZ에 있는 RDB로 빠르게 장애 복구를 할 수 있다. 가용성 및 내구성을 높여주므로 프로덕션 데이터베이스 워크로드에 적합하다.

- 파라미터 그룹: 파라미터 그룹이란 쉽게 말해 데이터베이스의 설정 옵션값을 말한다.
- 워크로드: 가장 일반적인 의미에서 워크로드는 시스템이나 네트워크가 작업을 완료하거나 특정 출력을 생성하는 데 걸리는 시간과 컴퓨팅 리소스의 양을 말한다.


**자동 백업**<br>
자동 백업, 줄여서 AB(Automated Backups)는 매일마다 스냅샷과 트랜잭션 로그를 참고하여 자동으로 백업을 해준다. RDS에서는 디폴트로 AB 기능이 설정되어있다. 다만, 롤백 동작은 과거 상태로 그대로 돌아가는 게 아닌, 다른 DB 인스턴스를 새로 생성해서 스냅샷을 적용 시키는 형식이다.

- 트랜잭션 로그: 데이터베이스에 수행되는 모든 변경 작업에 대한 기록을 갖는 특수한 파일 또는 메모리 영역이다.

**수동 백업**<br>
수동 백업은 유저 혹은 다른 프로세스로부터 요청에 따라 만들어지는 DB 스냅샷이다. 즉, EC2 스냅샷을 뜨듯이 사용자에 의해 수동적으로 진행되는 백업이다. 만약 원본 RDS를 삭제한다고 하더라도 스냅샷은 S3 버킷에 그대로 존재한다. 따라서 스냅샷만으로 RDS 인스턴스를 복원시킬 수 있다.

- S3: 온라인 오브젝트 스토리지 서비스이다.

**장점**
1. 간편한 관리
    - 인프라를 프로비저닝하거나 소프트웨어를 유지 관리할 필요 없이 비효율적이고 시간 소모적인 데이터베이스 관리 태스크를 제거할 수 있다.
2. 다양한 엔진 옵션
    - 선택한 관계형 데이터베이스 엔진을 클라우드 또는 온프레미스에 배포하고 규모를 조정할 수 있다.
3. 고가용성
    - Amazon RDS Multi-AZ 배포로 고가용성을 달성할 수 있다.
4. 운영 전문성
    - 클라우드에서 탄생한 데이터베이스에서 10년이 넘는 시간에 걸쳐 쌓익 검증된 운영 전문성, 보안 모범 사례 및 혁신을 활용할 수 있다.


# **CloudWatch**
CloudWatch는 애플리케이션을 모니터링하고, 성능 변화에 대응하고, 리소스 사용을 최적화하고, 운영 상태에 대한 인사이트를 제공하는 서비스이다. CloudWatch는 AWS 리소스 전반의 데이터를 수집하여 전체 시스템의 성능을 파악할 수 있도록 하고, 사용자가 경보를 설정하고 변화에 자동으로 대응하고, 운영 상태에 대한 통합된 뷰를 볼 수 있도록 한다.

**주요 기능**
1. 메트릭 수집 및 대시보드
    - 리소스의 성능 메트릭을 수집하고 사용자 지정 대시보드에 표시하여 모니터링 가능
2. 경보 및 자동 조치
    - 정의한 임계값을 기반으로 경고 및 자동 조치를 수행하는 경보 설정 가능
3. 이벤트
    - 리소스 상태 변경에 대한 이벤트를 모니터링하고 특정 이벤트에 대한 대응 조치 수행 가능
4. 로그 및 이벤트 데이터 분석
    - 로그 및 이벤트 데이터 수집 및 분석을 통해 리소스 동작 및 문제 분석 가능

# CloudTrail
AWS 계정 내에서 수행되는 모든 작업과 API 호출에 대한 감사 로그를 수집하고 보관한다. 계정 활동 모니터링, 보안 감사 및 규정 준수를 위해 사용된다.

**주요 기능**
1. API 호출 감사
    - AWS Management Console, CLI, SDK 등을 통한 모든 API 호출 및 작업 기록 감사
2. 이벤트 기록
    - 작업의 세부 정보, 타임스탬프, 사용자, IP 주소 등의 이벤트 정보 기록
3. 감사 로그 검색
    - 저장된 감사 로그를 쿼리하고 분석하여 특정 이벤트 추적 및 조사 가능
4. 다른 서비스와의 통합
    - CloudTrail 로그를 ClouWatch Logs, S3 등으로 전달하여 분석 및 장기 보존 가능
